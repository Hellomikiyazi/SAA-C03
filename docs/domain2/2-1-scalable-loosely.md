# Task 2.1：设计可扩展与松耦合架构

---

## 一、Task 考试地图

本 Task 的核心知识点按考试权重排列如下：

- SQS、SNS、EventBridge 的解耦模型与适用场景
- 标准队列与 FIFO 队列的顺序、吞吐量和去重差异
- Lambda、SQS、DynamoDB、S3 等 Serverless 组件组合实现弹性扩展
- Auto Scaling 的目标追踪、阶梯扩展、计划扩展与冷却时间
- ALB、NLB、API Gateway 在入口层扩展中的选择逻辑
- 无状态应用设计：会话外置、共享存储、水平扩展
- 缓存与读扩展：ElastiCache、CloudFront、DynamoDB DAX、RDS Read Replica
- 流式数据处理：Kinesis Data Streams、Kinesis Data Firehose、MSK

> **本 Task 最高频混淆对：** SQS（点对点队列） vs SNS（发布订阅广播）；EventBridge（事件总线/事件路由） vs SNS（消息扇出）；标准队列（高吞吐/可能乱序） vs FIFO 队列（严格顺序/去重）。

---

## 二、核心概念解析

### 1. 松耦合架构的本质

松耦合架构的核心是让系统组件之间**不要直接依赖彼此的可用性、吞吐能力和响应时间**。在紧耦合架构中，前端服务直接调用后端服务，后端慢或宕机时，请求会一路阻塞并放大故障。松耦合架构通常通过队列、事件总线、异步处理、缓存和托管服务隔离组件，使每一层都可以独立扩缩、独立失败、独立恢复。

最典型的考试场景是：Web 应用收到用户上传请求后，需要执行耗时任务（图片处理、订单履约、邮件发送、报表生成）。如果同步处理，用户请求会超时，流量高峰时后端会被打爆。正确设计是让 Web 层快速接收请求并把任务写入 SQS，后端 Worker 按自身能力慢慢消费队列。

> 考试判断：题目出现"突发流量导致后端处理不过来"、"请求超时"、"组件之间互相影响"、"需要异步处理" → 首先考虑 SQS 解耦。

---

### 2. Amazon SQS

Amazon SQS 是托管消息队列服务，用于在生产者和消费者之间建立缓冲层。生产者只负责把消息放入队列，消费者按自己的速度拉取消息处理。这样即使消费者短时间不可用，消息仍保存在队列中，不会直接丢失请求。

**SQS Standard Queue（标准队列）** 提供近乎无限吞吐量和至少一次投递（At-least-once Delivery）。它不保证严格顺序，少数情况下同一条消息可能被投递多次，因此消费者必须具备幂等性。标准队列适合绝大多数异步任务，例如图片处理、日志处理、订单后处理。

**SQS FIFO Queue（先进先出队列）** 提供严格顺序和恰好一次处理语义（通过去重机制避免重复处理）。FIFO 的吞吐量低于标准队列，适合顺序非常关键的场景，例如金融交易、库存扣减、按客户顺序处理事件。

**Visibility Timeout（可见性超时）** 是 SQS 的核心机制。消费者收到消息后，消息不会立即从队列删除，而是在一段时间内对其他消费者不可见。如果消费者处理成功，需要显式删除消息；如果处理失败或超时未删除，消息会重新变为可见，被其他消费者再次处理。

**Dead-Letter Queue（DLQ，死信队列）** 用于隔离多次处理失败的消息，避免坏消息在主队列中无限重试。考试中看到"消息重复失败，需要排查而不阻塞主流程"，答案通常是配置 DLQ。

> 考试判断：需要缓冲突发流量、异步处理、削峰填谷 → SQS；需要严格顺序 → SQS FIFO；需要处理失败消息隔离 → SQS DLQ；看到 SQS 消费者可能重复处理 → 应用必须幂等。

---

### 3. Amazon SNS

Amazon SNS 是发布订阅（Pub/Sub）消息服务。发布者将消息发送到 Topic，SNS 再把消息推送给所有订阅者。订阅者可以是 SQS、Lambda、HTTP/S 端点、Email、SMS 等。

SNS 最核心的价值是**广播和扇出（Fanout）**。例如订单系统创建订单后，需要同时触发库存系统、发票系统、邮件系统和数据分析系统。订单服务不应该分别调用四个系统，而是发布一条事件到 SNS Topic，由 SNS 同时推送给多个订阅者。这样新增订阅者不会影响订单服务。

SNS 与 SQS 经常组合使用：SNS 负责广播，SQS 负责为每个消费者提供独立缓冲。每个下游系统订阅自己的 SQS 队列，即使某个系统处理慢，也不会影响其他系统。

> 考试判断：一条消息需要通知多个下游系统 → SNS；每个下游系统还需要独立重试和缓冲 → SNS + 多个 SQS 队列。

---

### 4. Amazon EventBridge

Amazon EventBridge 是事件总线服务，用于基于事件规则进行路由、过滤和集成。它可以接收 AWS 服务事件、自定义应用事件、SaaS 应用事件，然后根据规则把事件发送到 Lambda、SQS、SNS、Step Functions、Kinesis 等目标。

EventBridge 与 SNS 都能做事件分发，但定位不同。SNS 更像高吞吐的消息广播服务，重点是把消息推送给订阅者；EventBridge 更像事件路由和集成平台，重点是按事件内容、来源、字段规则进行过滤和转发，并支持大量 AWS/SaaS 事件源。

典型考试场景：当 EC2 实例状态变为 stopped 时触发 Lambda；当 S3 对象创建事件发生时启动 Step Functions；从第三方 SaaS 接收事件并路由到内部处理系统。这类"根据事件模式触发动作"的题目，EventBridge 通常是标准答案。

> 考试判断：题目强调"事件驱动"、"按事件模式过滤"、"AWS 服务状态变化触发"、"SaaS 事件集成" → EventBridge。

---

### 5. Auto Scaling 与水平扩展

可扩展架构的核心是水平扩展（Scale Out），即通过增加更多实例处理更多流量，而不是只升级单台机器规格。AWS 中最典型的组合是 ALB + Auto Scaling Group + 多 AZ EC2。

**目标追踪扩展（Target Tracking）** 是最常见的策略，例如让 ASG 始终保持平均 CPU 使用率 50%。它简单、自动、适合大多数负载。

**阶梯扩展（Step Scaling）** 根据告警严重程度执行不同扩容幅度，例如 CPU 超过 70% 增加 2 台，超过 90% 增加 5 台。适合需要更精细控制的场景。

**计划扩展（Scheduled Scaling）** 用于可预测的流量高峰，例如每天早上 9 点业务流量上涨，提前扩容避免冷启动。

**冷却时间（Cooldown）** 防止扩缩容动作过于频繁。扩容后系统需要时间让新实例启动并接收流量，冷却时间可以避免 Auto Scaling 因短时指标波动反复扩缩。

> 考试判断：不可预测负载 → Target Tracking；流量峰值时间固定 → Scheduled Scaling；需要按严重程度分级扩容 → Step Scaling。

---

### 6. 无状态应用设计

水平扩展要求应用实例尽量无状态。无状态意味着任意请求可以被负载均衡器发送到任意实例，而不会因为用户会话、临时文件或本地缓存只存在于某台实例上导致失败。

**会话状态** 不应保存在 EC2 本地内存中，而应外置到 ElastiCache、DynamoDB 或 RDS。这样实例被替换、扩容或缩容时，用户会话不会丢失。

**用户上传文件** 不应保存在 EC2 本地磁盘，而应保存到 S3；多个 EC2 需要共享文件系统时，可以使用 EFS。EBS 适合单个 EC2 的块存储，不适合作为多实例共享文件系统。

**配置和密钥** 不应写死在实例中，而应放在 Systems Manager Parameter Store、Secrets Manager 或环境变量中，便于新实例自动拉取配置。

> 考试判断：题目出现"用户会话丢失"、"负载均衡后请求必须回到同一台实例" → 将 Session 外置到 ElastiCache/DynamoDB；出现"多台 EC2 共享文件" → EFS；出现"静态资源或上传文件" → S3。

---

### 7. 入口层扩展：ALB、NLB、API Gateway

**Application Load Balancer（ALB）** 工作在第 7 层，理解 HTTP/HTTPS，可以基于路径、Host Header、HTTP Header 做路由，适合 Web 应用、微服务、容器服务。考试中只要是 HTTP 应用负载均衡，大多数情况下优先选 ALB。

**Network Load Balancer（NLB）** 工作在第 4 层，面向极高性能、超低延迟、TCP/UDP/TLS 流量，并支持静态 IP。适合非 HTTP 协议、需要固定入口 IP、极高吞吐的场景。

**API Gateway** 是托管 API 入口，常与 Lambda 组合构建 Serverless API。它提供认证授权、限流、缓存、请求转换、API Key、阶段部署等能力。相比 ALB，API Gateway 更适合对外发布 API 和 Serverless 后端；相比 NLB，它不用于通用 TCP/UDP 负载均衡。

> 考试判断：HTTP 路径路由/微服务 → ALB；TCP/UDP/静态 IP/极低延迟 → NLB；Serverless API、限流、API Key、请求转换 → API Gateway。

---

### 8. 缓存与读扩展

缓存是扩展读性能、降低后端压力的重要手段。SAA 考试通常根据缓存位置和数据类型来选择服务。

**CloudFront** 是边缘缓存，适合缓存静态内容、视频、图片、前端资源，也可以加速动态内容。它把内容分发到全球边缘节点，减少用户到源站的延迟。题目出现"全球用户访问静态网站慢"、"降低 S3/ALB 源站压力"，优先考虑 CloudFront。

**ElastiCache for Redis/Memcached** 是内存缓存，位于应用和数据库之间，适合缓存热点查询结果、会话状态、排行榜、令牌等低延迟数据。Redis 支持更丰富的数据结构、持久化、复制和高可用，是考试中更常见的答案。

**DynamoDB DAX** 是 DynamoDB 专用的全托管内存缓存，可将读取延迟从毫秒级降到微秒级。它只适合 DynamoDB，不适合 RDS 或通用缓存。

**RDS Read Replica** 用于关系型数据库读扩展，将读请求分流到只读副本。它不负责自动故障切换，高可用相关细节在 Task 2.2 中会重点区分。

> 考试判断：全球静态内容加速 → CloudFront；数据库热点查询缓存 → ElastiCache；DynamoDB 微秒级读缓存 → DAX；关系型数据库读多写少 → Read Replica。

---

### 9. Kinesis 与流式处理

Kinesis 用于处理持续产生的流式数据，例如点击流、IoT 传感器数据、应用日志、实时指标。

**Kinesis Data Streams** 适合需要自定义消费者、实时处理、多个消费者重复读取同一数据流的场景。数据进入 Stream 后按 Shard 分区保存，消费者可以用 Lambda、Kinesis Client Library 或自定义应用读取处理。它强调实时处理和可重放。

**Kinesis Data Firehose** 适合把流式数据近实时投递到 S3、Redshift、OpenSearch、第三方 HTTP 端点。Firehose 是全托管交付服务，不需要管理消费者，也不强调复杂实时处理逻辑。

**Amazon MSK（Managed Streaming for Apache Kafka）** 适合已有 Kafka 生态、需要 Kafka API 兼容、复杂事件流平台的场景。考试中如果题目明确提到迁移 Kafka 工作负载，MSK 通常是答案。

> 考试判断：实时自定义消费/多消费者/可重放 → Kinesis Data Streams；直接把流数据送到 S3/Redshift/OpenSearch → Kinesis Data Firehose；Kafka 兼容 → MSK。

---

### 10. Step Functions 与编排

AWS Step Functions 用于编排多步骤工作流。它可以把 Lambda、ECS、Batch、DynamoDB、SNS、SQS 等服务串联成状态机，并内置重试、错误处理、分支、并行、等待等控制逻辑。

它解决的问题不是简单消息解耦，而是**有状态流程编排**。例如订单处理需要依次完成支付、库存扣减、发票生成、通知用户，中间任一步失败都要重试或走补偿逻辑。把这些逻辑硬编码在 Lambda 中会复杂且难维护，Step Functions 能让流程可视化并托管执行状态。

> 考试判断：题目出现"多个步骤"、"流程编排"、"状态跟踪"、"错误重试和补偿逻辑" → Step Functions。

| 通信模型 | 队列，点对点消费 | 发布订阅，广播推送 |

## 四、高频复合场景

**场景一：电商网站在秒杀活动中订单请求暴增，后端库存服务处理不过来，用户请求频繁超时。**

推理路径：前端请求量短时间暴增，后端处理能力有限，核心问题是同步调用导致压力直接传递。应在 Web 层和库存处理层之间加入 SQS 队列，Web 层快速写入订单消息并返回响应，后端 Worker 从队列中按能力消费。Auto Scaling 可以根据 SQS 队列深度扩展 Worker 数量。如果库存扣减必须严格按商品或订单顺序处理，则选择 SQS FIFO；如果顺序不重要，选择标准队列获得更高吞吐。

最终答案：ALB + Auto Scaling Web 层 + SQS 解耦 + Worker Auto Scaling；顺序敏感时使用 SQS FIFO。

**场景二：订单创建后需要同时触发发票、邮件、物流和数据分析系统，未来还可能新增更多下游消费者。**

推理路径：一条业务事件需要通知多个系统，且新增消费者不应修改订单服务。SNS Topic 是发布订阅模型的标准选择。为了让每个下游系统独立缓冲和重试，应让每个系统订阅自己的 SQS 队列，而不是让 SNS 直接调用所有服务。

最终答案：订单服务发布到 SNS Topic，SNS Fanout 到多个 SQS 队列，各系统独立消费。

**场景三：公司希望在 EC2 实例进入 stopped 状态时自动触发 Lambda 创建工单，并只处理特定标签的实例。**

推理路径：这是 AWS 服务状态变化触发自动动作，且需要按事件内容过滤。EventBridge 能匹配 EC2 状态变化事件，并根据标签或事件字段规则路由到 Lambda。SNS 虽然可以通知，但不是这类 AWS 事件规则匹配的最佳答案。

最终答案：EventBridge 规则匹配 EC2 状态变化事件，目标设置为 Lambda。

**场景四：Web 应用部署在多台 EC2 后，用户登录状态经常丢失，必须开启粘性会话才勉强可用。**

推理路径：问题根因是 Session 存在单台 EC2 本地内存中，导致请求被 ALB 分发到其他实例时无法识别用户状态。粘性会话只是缓解，不是可扩展架构的最佳实践。应将 Session 外置到 ElastiCache Redis 或 DynamoDB，使任意实例都能读取会话状态。

最终答案：应用无状态化，将 Session 存储到 ElastiCache Redis 或 DynamoDB，继续使用 ALB 跨实例分发请求。

**场景五：全球用户访问 S3 中的图片和前端静态资源很慢，同时源站请求费用和流量压力较高。**

推理路径：问题是全球用户到源站距离远，以及重复请求直接打到 S3。CloudFront 可以把静态内容缓存到边缘节点，用户就近访问，降低延迟并减少 S3 源站压力。如果需要保护私有内容，可以结合 Origin Access Control 和签名 URL。

最终答案：S3 作为源站，前面加 CloudFront 分发静态内容。

**场景六：IoT 设备持续发送传感器数据，应用需要实时分析，同时多个消费者需要读取同一批数据。**

推理路径：这是持续流式数据，不是普通异步任务队列。多个消费者需要独立读取和处理同一数据流，且需要实时分析和可重放，Kinesis Data Streams 更合适。Firehose 更适合直接投递到 S3/Redshift/OpenSearch，不适合作为复杂实时消费平台。

最终答案：Kinesis Data Streams 接收数据，Lambda 或自定义消费者实时处理；需要长期存储时再投递到 S3。

**场景七：一个订单流程包含支付、库存、配送、通知多个步骤，每一步都可能失败并需要重试或补偿。**

推理路径：这不是单纯的队列缓冲，而是多步骤工作流编排。Step Functions 可以定义状态机，托管每一步的执行状态、重试策略、错误分支和补偿路径。SQS 可以作为某些异步步骤的缓冲，但不能替代整体流程编排。

最终答案：Step Functions 编排订单工作流，必要步骤调用 Lambda/SQS/SNS/DynamoDB。

---

## 五、本 Task 一句话总结

- **SQS = 解耦和缓冲**，看到异步处理、削峰填谷、后端处理慢，优先想到 SQS。
- **SNS = 广播通知**，一条消息要触发多个下游系统时，用 SNS Fanout，常与多个 SQS 队列组合。
- **EventBridge = 事件路由**，看到 AWS 服务事件、SaaS 集成、按事件模式过滤，优先选 EventBridge。
- **可水平扩展的前提是无状态**，Session 放 ElastiCache/DynamoDB，文件放 S3/EFS，实例可以随时替换。
- **缓存选择看位置**：全球边缘缓存用 CloudFront，应用侧热点缓存用 ElastiCache，DynamoDB 专用微秒级缓存用 DAX。
