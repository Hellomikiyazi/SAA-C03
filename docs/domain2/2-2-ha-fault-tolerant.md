# Task 2.2：设计高可用与容错架构

---

## 一、Task 考试地图

本 Task 的核心知识点按考试权重排列如下：

- 四种灾难恢复（DR）策略的 RPO/RTO 与成本权衡
- Multi-AZ 与 Read Replica 的区别与适用场景
- Route 53 路由策略（Failover、加权、延迟、地理位置）
- Auto Scaling 与 ELB 组合实现弹性高可用
- 单点故障（SPOF）的识别与消除
- RDS Proxy 的作用与适用场景
- 数据持久性策略（S3 跨区域复制、EBS 快照、备份自动化）

> **本 Task 最高频混淆对：** Multi-AZ（高可用/容灾） vs Read Replica（读性能扩展）；Pilot Light vs Warm Standby（DR 策略的成本与恢复速度权衡）。

---

## 二、核心概念解析

### 1. RPO 与 RTO（灾难恢复的两把尺子）

理解 DR 策略的前提是先搞清楚这两个概念的含义，因为它们是考题中筛选答案的核心依据。

**RPO（Recovery Point Objective，恢复点目标）** 回答的是"最多可以丢失多少数据"。如果 RPO = 1 小时，意味着系统每小时备份一次，最坏情况下会丢失 1 小时的数据。RPO 越小，说明对数据丢失的容忍度越低，所需的备份频率越高，成本也越高。

**RTO（Recovery Time Objective，恢复时间目标）** 回答的是"灾难发生后最多需要多久恢复服务"。RTO = 4 小时意味着系统最多允许中断 4 小时。RTO 越小，说明对停机时间的容忍度越低，需要在待命状态中维持更多资源，成本也越高。

> 考试判断：题目中出现"数据丢失不能超过 X 小时"→ 看 RPO；出现"服务中断不能超过 X 分钟"→ 看 RTO。两者都要求极低时，答案通常是 Active-Active 架构或 Multi-AZ。

---

### 2. 四种灾难恢复策略

这四种策略构成一个从"便宜但慢"到"贵但快"的连续谱，考试题目通常给出 RPO/RTO 要求和预算约束，你需要选出最合适的那一种。

**Backup and Restore（备份与恢复）** 是成本最低的方案。做法是定期将数据快照备份到 S3，灾难发生时从备份重新创建所有资源。RPO 和 RTO 都是最高的（通常以小时计），因为恢复时需要从头启动所有基础设施。它的价值在于几乎零平时运营成本，适合对恢复时间没有严格要求的非关键系统。

**Pilot Light（飞行员灯）** 的核心思想来自热水器的长明导火灯——平时只保持最关键的核心组件运行（通常是数据库持续同步），其他计算资源处于关闭状态。灾难发生时，快速启动应用层资源，连接已经同步好的数据库。与 Backup/Restore 相比，RTO 缩短到分钟级，因为不需要等待数据恢复，只需启动计算资源。但平时要持续为运行中的数据库付费。

**Warm Standby（温备份）** 是 Pilot Light 的加强版——不仅核心组件在运行，整个系统的缩小版（小规格实例）也在运行中。灾难发生时，只需将缩小版扩展到生产规格即可，RTO 进一步缩短。平时维护成本高于 Pilot Light，但低于 Active-Active。

**Active-Active（双活/多活）** 是 RPO ≈ 0、RTO ≈ 0 的终极方案。两个或多个区域同时承载生产流量，任意区域故障时，其他区域直接接管，几乎无感知。Route 53 通过健康检查自动切换流量。成本最高，因为需要在多个区域同时维持完整的生产级资源。

| DR 策略 | RPO | RTO | 平时成本 | 适用场景 |
|--------|-----|-----|---------|---------|
| Backup & Restore | 高（小时级） | 高（小时级） | 最低 | 非关键系统 |
| Pilot Light | 中（分钟级） | 中（分钟级） | 低 | 核心数据不能丢 |
| Warm Standby | 低（秒~分钟） | 低（分钟级） | 中 | 关键业务系统 |
| Active-Active | 极低（接近 0） | 极低（接近 0） | 最高 | 零停机容忍 |

### 3. RDS Multi-AZ 与 Read Replica

这是本 Task 中最高频的混淆考点，两者在名字和功能上都有相似之处，但设计目的完全不同。

**RDS Multi-AZ** 的设计目的是**高可用（HA）**，不是性能提升。AWS 在另一个 AZ 同步维护一个待机副本（Standby），应用程序始终只连接主实例，待机副本不接受任何读写请求。当主实例发生故障时，AWS 自动将 DNS 切换到待机副本，通常在 1-2 分钟内完成，应用程序无需修改代码（连接字符串不变）。Multi-AZ 是一种保险机制，平时对性能没有任何帮助，但关键时刻保障服务连续性。

**Read Replica（只读副本）** 的设计目的是**读性能扩展**，不是容灾。通过异步复制将数据同步到只读副本，应用程序需要将读请求显式指向 Read Replica 的端点。Read Replica 可以跨 Region 部署，也可以被提升为独立的主实例（手动操作），但这个提升过程不是自动的，也不像 Multi-AZ 那样有托管的故障切换机制。

> 考试判断：题目出现"数据库主节点宕机自动恢复"→ Multi-AZ；出现"读操作压力过大需要分担"→ Read Replica；出现"跨区域灾备"→ Read Replica（可跨 Region）+ 手动提升。注意：Multi-AZ 与 Read Replica 可以同时使用，互不排斥。

---

### 4. Route 53 路由策略

Route 53 是 AWS 的 DNS 服务，其路由策略是实现多区域高可用和流量管理的关键工具。

**Failover 路由（故障转移）** 专为主备架构设计。配置主记录（Primary）和备用记录（Secondary），Route 53 持续对主端点做健康检查，一旦主端点不健康，自动将所有流量切换到备用端点。这是实现跨 Region DR 的标准做法。

**加权路由（Weighted）** 按比例分配流量到多个端点（如 70% 到 us-east-1，30% 到 eu-west-1），适合蓝绿部署或 A/B 测试场景中的渐进式流量迁移。

**延迟路由（Latency-based）** 将用户请求路由到与用户网络延迟最低的 AWS Region，适合全球用户访问、需要最佳响应速度的场景。

**地理位置路由（Geolocation）** 按用户的实际地理位置（国家/洲）路由，适合需要合规隔离（如欧盟用户数据不能离开欧洲）或提供本地化内容的场景。

> 考试判断："主备自动切换"→ Failover；"渐进式流量迁移/蓝绿部署"→ Weighted；"全球用户最低延迟"→ Latency-based；"按地区合规隔离"→ Geolocation。

---

### 5. Auto Scaling 与 ELB 的协同

**Elastic Load Balancing（ELB）** 将入站流量分发到多个目标（EC2 实例、容器、Lambda），同时对目标做健康检查，自动停止向不健康的实例发送流量。ELB 本身跨多个 AZ 部署，是消除前端单点故障的标准方案。

**Auto Scaling** 根据需求自动增减 EC2 实例数量。核心概念包括：**最小容量**（保证基本可用性）、**最大容量**（控制成本上限）、**期望容量**（正常运行时的目标数量）。扩展策略分为目标追踪（Target Tracking，如保持 CPU 在 50%）、阶梯扩展（Step Scaling）和计划扩展（Scheduled，适合可预期的流量高峰）。

ELB 与 Auto Scaling 组合在一起构成 AWS 高可用的"黄金搭档"：ELB 负责流量分发和健康检查，Auto Scaling 负责实例数量的弹性调整，两者都跨多 AZ 部署，共同消除单点故障。

---

### 6. RDS Proxy

RDS Proxy 是一个数据库代理层，位于应用程序和 RDS/Aurora 数据库之间，解决两个核心问题。

**连接池管理**：Serverless 应用（如 Lambda）的特点是瞬间可以创建大量并发执行，每个执行都需要独立的数据库连接，容易导致数据库连接数耗尽（Connection Exhaustion）。RDS Proxy 在应用层和数据库之间维护一个连接池，将大量来自 Lambda 的短连接复用为少量到数据库的长连接，大幅降低数据库的连接压力。

**故障转移加速**：在 Multi-AZ 切换时，RDS Proxy 可以将故障转移时间从约 60 秒缩短到约 20 秒，因为代理层会维护连接状态，切换对应用程序更透明。

> 考试判断：题目出现 Lambda 大规模并发访问 RDS 导致连接超限→ RDS Proxy 是标准答案。

---

### 7. 单点故障（SPOF）的识别与消除

单点故障是指系统中一旦失效就会导致整体服务中断的单个组件。SAA 考试中经常出现"现有架构的问题"类题型，要求识别 SPOF 并给出修复方案。

常见 SPOF 及对应消除方案：单台 EC2 → Multi-AZ + Auto Scaling + ELB；单个 AZ 的 NAT Gateway → 每个 AZ 部署独立的 NAT Gateway；单 AZ 的 RDS → 启用 Multi-AZ；单区域数据存储 → S3 跨区域复制（CRR）或 RDS 跨区域 Read Replica；静态 IP 绑定的服务入口 → 改用 ELB + Route 53 DNS 名称。

## 三、服务对比速查表

| 对比维度 | RDS Multi-AZ | RDS Read Replica |
|---------|-------------|-----------------|
| 设计目的 | 高可用（HA） | 读性能扩展 |
| 复制方式 | 同步 | 异步 |
| 待机副本可读 | ❌ 不可读 | ✅ 可读 |
| 故障自动切换 | ✅ 自动（1-2 分钟） | ❌ 手动提升 |
| 跨 Region 支持 | ❌ 同 Region 跨 AZ | ✅ 支持跨 Region |
| 适用考试场景 | 主库宕机自动恢复 | 读压力分担 / 跨区域灾备 |

| 对比维度 | Pilot Light | Warm Standby |
|---------|------------|-------------|
| 平时运行内容 | 只有核心数据层（如 DB 同步） | 完整系统的缩小版 |
| RTO | 分钟级（需启动应用层） | 更短（只需扩容，无需启动） |
| 平时成本 | 低 | 中 |
| 恢复操作 | 启动 + 配置应用资源 | 扩展实例规格 |

| 对比维度 | Route 53 Failover | Route 53 Latency-based |
|---------|------------------|----------------------|
| 路由依据 | 健康检查结果 | 用户到各 Region 的网络延迟 |
| 主要用途 | 主备切换 / 灾难恢复 | 全球用户就近访问 |
| 是否需要健康检查 | ✅ 必须配置 | 可选 |

## 四、高频复合场景

**场景一：全球电商平台，要求任意 Region 故障后 5 分钟内恢复，数据不能丢失。**

推理路径：RTO ≤ 5 分钟且数据零丢失，排除 Backup/Restore 和 Pilot Light（RTO 过长）。在 Warm Standby 和 Active-Active 之间，因为"5 分钟内恢复"且"数据不能丢失"，Active-Active 是更稳妥的答案；Route 53 Failover + 健康检查实现自动切换；Aurora Global Database 实现跨 Region 数据同步（复制延迟 < 1 秒）。

**场景二：Lambda 函数每天特定时段触发大量并发执行，频繁出现数据库连接超时错误。**

推理路径：Lambda 并发 → 大量短连接 → RDS 连接数耗尽，标准解法是在 Lambda 和 RDS 之间插入 RDS Proxy，由 Proxy 维护连接池。同时考虑开启 RDS Multi-AZ 保障数据库本身的高可用性。

**场景三：一家医疗公司要求数据库备份保留 7 年且不可删除，即使管理员也不能提前删除。**

推理路径：长期不可删除 → AWS Backup 配合 Vault Lock（WORM 模式）；Vault Lock 启用 Compliance Mode 后，连 root 账号也无法在保留期届满前删除备份，满足合规要求。

**场景四：应用层 EC2 在一个 AZ 中，数据库 RDS 也在同一 AZ，要以最小改动提升整体可用性。**

推理路径：两个 SPOF——EC2 单 AZ 和 RDS 单 AZ。最小改动方案：① 启用 RDS Multi-AZ（一键开启，无需修改应用代码）；② 在两个 AZ 各部署 EC2，前端添加 ALB；③ 确保 Auto Scaling Group 跨两个 AZ 分布。这是"渐进式改造遗留系统"的标准考题答案路径。

**场景五：需要在 us-east-1 主区域故障时，将流量自动切换到 eu-west-1，且切换前后终端用户无需修改 DNS 配置。**

推理路径：跨 Region 自动切换 → Route 53 Failover 路由策略；主端点配置 Health Check，一旦 us-east-1 的 ALB 健康检查失败，Route 53 自动将同一域名解析指向 eu-west-1 的备用端点；用户使用的域名始终不变，满足"无需修改配置"的要求。

---

## 五、本 Task 一句话总结

- **DR 策略选择**：RPO/RTO 要求越严格、成本预算越高 → 从 Backup/Restore 往 Active-Active 方向走。
- **Multi-AZ = 容灾，Read Replica = 读扩展**，两者目的不同，不可混用来回答对方的场景。
- **Route 53 Failover + 健康检查** 是跨 Region 自动切换的标准组合，Latency-based 解决全球就近访问。
- **Lambda + RDS 出现连接超时**，第一反应是 RDS Proxy，而不是升级 RDS 实例规格。
- **消除 SPOF 的三板斧**：多 AZ 部署 + ELB + Auto Scaling，这个组合几乎适用于所有 EC2 高可用场景。
