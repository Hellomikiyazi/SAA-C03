### Question #1

Topic 1


A company collects data for temperature, humidity, and atmospheric pressure in cities across multiple continents. The average volume of data

that the company collects from each site daily is 500 GB. Each site has a high-speed Internet connection.

The company wants to <u>aggregate the data from all these global sites as quickly as possible in a single Amazon S3 bucket</u>. The solution must

<u>minimize operational complexity(尽量减少运维难度)</u>.

Which solution meets these requirements?


A. Turn on S3 Transfer Acceleration on the destination S3 bucket. Use multipart uploads to directly upload site data to the destination S3

bucket.(开启目标 S3 bucket的 S3 Transfer Acceleration， 然后使用 multipart uploads 让各个站点直接把数据上传到目标S3 bucket)

- S3 Transfer Acceleration 会利用 AWS 全球边缘站点，把全球各地的数据先上传到离站点较劲的 AWS edge location，然后通过AWS 的优化网络传输到目标S3 Bucket所在的Region。

- 数据直接进入最终目标的S3 bucket，不需要中转bucket和EC、EBS、快照；全球上传速度更快，运维难度最低。

B. Upload the data from each site to an S3 bucket in the closest Region. Use S3 Cross-Region Replication to copy objects to the

destination S3 bucket. Then remove the data from the origin S3 bucket.

- 从每个站点先把数据上传到离自己最近的AWS Region 的S3 bucket，然后在使用 S3 Cross-Region Replication（CRR）把数据复制到最终 S3 bucket，在从源S3 bucket 删除数据。

- 这个方案也是可行的，但不是最快运维最简单。

C. Schedule AWS Snowball Edge Storage Optimized device jobs daily to transfer data from each site to the closest Region. Use S3 Cross-

Region Replication to copy objects to the destination S3 bucket.

- 每天为每个站点安排 AWS Snowball Edge Storage Optimized，把数据传到最近的Region，然后再通过 S3 Cross-Region Replication 复制到目标 S3 bucket。

- Snowball Edge Storage 适合 网络带宽不足，一次性迁移大量数据，离线或者半离线传输，PB级别数据迁移。


D. Upload the data from each site to an Amazon EC2 instance in the closest Region. Store the data in an Amazon Elastic Block Store

(Amazon EBS) volume. At regular intervals, take an EBS snapshot and copy it to the Region that contains the destination S3 bucket.

Restore the EBS volume in that Region.

- EBS 是EC2的持久块存储

Correct Answer: A

- aggregate 汇总
- S3是高持久性、高扩展性的对象存储；S3 bucket 是 AWS S3 的对象存储容器；S3 Transfer Acceleration 是让全球客户端更快地把数据上传到这个bucket的网络加速能力。






### Question #2

Topic 1

A company needs the ability to analyze the log files of its proprietary application. The logs are stored in JSON format in an Amazon S3

bucket. Queries will be simple and will run on-demand. A solutions architect needs to perform the analysis with <u>minimal changes</u> to the

existing architecture.

- S3 已经有JSON结构的数据，想用SQL做临时查询，且要求最少运维开销是，选择 Amazon Athena。

What should the solutions architect do to meet these requirements with the LEAST amount of operational overhead?

A. Use Amazon Redshift to load all the content into one place and run the SQL queries as needed.

- RedShift 适合用于高并发的BI；复杂分析

B. Use Amazon CloudWatch Logs to store the logs. Run SQL queries as needed from the Amazon CloudWatch console.

- CloudWatch 可以用来查询日志，但是使用 CloudWatch Logs Insights 的查询语言，不是标准的SQL。现在日志在S3上，使用CloudWatch Logs 会改变现有的架构。

- CloudWatch 适合分析应用运行时日志，指标，告警和近实时运维日志。

C. Use Amazon Athena directly with Amazon S3 to run the queries as needed.

- Athena 是 serverless 查询服务，可以直接对S3中的数据运行SQL查询，支持 JSON，CSV， Parquet和ORC等格式。

- 通常还可以配合AWS Glue Data Catalog 定义表结构。

### Question #3

Topic 1

A company uses AWS Organizations to manage multiple AWS accounts for different departments. The management account has an Amazon

S3 bucket that contains project reports. The company wants to limit access to this S3 bucket to only users of accounts within the organization

in AWS Organizations.

- 考点：如何用 S3 bucket policy 限制访问来源必须属于同一个AWS Organization。
- 分析：
  - 公司使用 AWS Organization 管理多个AWS账户；
  - management accout 里面只有一个 S3 bucket
  - 只允许 organization 内部的账户的用户访问这个bucket
  - 需要 least operational overhead
- 这基本指向：PrincipalOrgID

Which solution meets these requirements with the LEAST amount of operational overhead?

A. Add the aws PrincipalOrgID global condition key with a reference to the organization ID to the S3 bucket policy.

- PrincipalOrgID 是什么
  - 它是IAM policy里面一个 global condition key，可以判断请求方 principal 所属的 AWS account 是否属于指定的 AWS Organization。
- 在 S3 bucket policy 里面添加aws：PrincipalOrgID 条件，并指向organization ID 。

B. Create an organizational unit (OU) for each department. Add the aws:PrincipalOrgPaths global condition key to the S3 bucket policy.
- 为每个部门创建OU，在 S3 bucket policy 里面添加aws：PrincipalOrgPaths。这个条件的颗粒度更细，适合限制某些OU或者某些部门的访问。

C. Use AWS CloudTrail to monitor the CreateAccount, InviteAccountToOrganization, LeaveOrganization, and

RemoveAccountFromOrganization events. Update the S3 bucket policy accordingly.

- 用 CloudTrail 监控账号的创建，要求，离开组织，移除组织等时间，根据这些事件动态更新S3 bucket policy。
- 这个方案是可行的，但是运维的复杂程度更高。
  - 需要主动进行监控 CloudTrail 事件；
  - 出发 Lambda 或者自动化流程；
  - 维护 accountID 列表；
  - 更新bucket policy；
  - 处理异常和同步延迟。

D. Tag each user that needs access to the S3 bucket. Add the aws:PrincipalTag global condition key to the S3 bucket policy.
- 给每个需要访问 S3 bucket 的用户打tag，然后用到aws:PrincipalTag
- 这个方案不适合。
- aws:PrincipalTag 适合基于身份标签做 ABAC（Attribute-Based Access Control）
- 给每个用户打标签，维护成本也是很高的。

Correct Answer: A

### Question #5

Topic 1

A company is hosting a web application on AWS using a single Amazon EC2 instance that stores user-uploaded documents in an Amazon EBS

volume. For better scalability and availability, the company duplicated the architecture and created a second EC2 instance and EBS volume in

another Availability Zone, placing both behind an Application Load Balancer. After completing this change, users reported that, each time they

refreshed the website, they could see one subset of their documents or the other, but never all of the documents at the same time.

What should a solutions architect propose to ensure users see all of their documents at once?

- 考点：多台 EC2 跨 AZ 共享文件时，应该使用Amazon EFS，而不是使用各自本地的EBS。

原来的架构：

```text
 User
 ↓
EC2 instance
 ↓
EBS volume：存用户上传的 documents
```

后来为了提高可扩展性和可用性，公司做了第二套架构：

```text
                 ┌─ EC2 A → EBS A
User → ALB ──────┤
                 └─ EC2 B → EBS B
```
两个 EC2 在不同 AZ，并且防止 ALB 后面。
出现了问题：用户刷新网页，用户只能看到两个不同文件的子集，而不是完整的合集。
  
**根本原因：EBS 绑定到单个AZ，单个EC2使用的块存储，不适合多台EC2共享的文件系统**

A. Copy the data so both EBS volumes contain all the documents

- 这不是一个好的方案，因为只能解决一次性的历史数据问题，不能解决之后的文件一致性问题。

B. Configure the Application Load Balancer to direct a user to the server with the documents

- 配置ALB，让用户总是被转发到其他 documents 的那台服务器上。这个不是正确的架构。不能解决真正解决共享存储的问题。 

C. Copy the data from both EBS volumes to Amazon EFS. Modify the application to save new documents to Amazon EFS

- EFS 是托管的共享文件系统，可以同时被多个EC2实例挂载，并且可跨多个AZ使用。
修改后的架构：

```text
                 ┌─ EC2 A ─┐
User → ALB ──────┤         ├→ Amazon EFS
                 └─ EC2 B ─┘
```

D. Configure the Application Load Balancer to send the request to both servers. Return each document from the correct server
- ALB 是负载均衡器，会把一个请求路由到一个目标实例。而不是把同一个请求同时发给所有服务器并合并结果。

Correct Answer: C

- 考点：
  - ALB 解决流量分发，不能解决数据共享；
  - EBS 适合单台实例持久块存储
  - EFS 适合多实例共享文件存储



### Question #6

Topic 1

A company uses NFS to store large video files in on-premises（现场） network attached storage. Each video file ranges in size from 1 MB to 500 GB.

The total storage is 70 TB and is no longer growing. The company decides to migrate the video files to Amazon S3. The company must

migrate the video files as soon as possible while using the least possible network bandwidth.

Which solution will meet these requirements?

- 公司本地有一套 NFS 网络存储，存放了大量的视频文件，总量为70GB。总量不会再增长了，希望迁移到 AWS S3，而且不用占用带宽.

A. Create an S3 bucket. Create an IAM role that has permissions to write to the S3 bucket. Use the AWS CLI to copy all files locally to the

S3 bucket.

- 使用 CLI 复制到 S3 是可行的，但是会占用大量的带宽。 CLI 适合 小量或中等规模的数据上传，也适合增量数据、自动花脚本上传。

B. Create an AWS Snowball Edge job. Receive a Snowball Edge device on premises. Use the Snowball Edge client to transfer data to the

device. Return the device so that AWS can import the data into Amazon S3.

- SnowBall Edge 是 AWS寄一个物理设备到本地，公司把 NFS中的数据复制到里面，再把 SnowBall Edge 机会去 AWS。AWS倒入 S3.
- 优势是避免通过网络传输大规模数据，适合一次性迁移。

C. Deploy an S3 File Gateway on premises. Create a public service endpoint to connect to the S3 File Gateway. Create an S3 bucket.

Create a new NFS file share on the S3 File Gateway. Point the new file share to the S3 bucket. Transfer the data from the existing NFS file

share to the S3 File Gateway.

- S3 File Gateway 可以让 本地应用 通过 NFS 或者 SMB 访问 S3，为的是混合云场景，比如本地应用继续用文件协议，而后端数据落在 S3.

D. Set up an AWS Direct Connect connection between the on-premises network and AWS. Deploy an S3 File Gateway on premises. Create a

public virtual interface (VIF) to connect to the S3 File Gateway. Create an S3 bucket. Create a new NFS file share on the S3 File Gateway.

Point the new file share to the S3 bucket. Transfer the data from the existing NFS file share to the S3 File Gateway.

- Direct Connect 使用的专线连接，适合长期稳定、低延迟、高吞吐的混合云链接。

Correct Answer: B

#### 这一题的判断口诀
1. Snowball Edge： 几十/几百 B TB 数据 + 一次性迁移 + 本地数据中心 + 尽快 + 减少网络带宽
2. AWS Storage Gateway / S3 File Gateway：持续访问本地文件协议（Continuous access to local files protocol）、NFS/SMB、把本地应用接到 S3、缓存热数据
3. AWS Direct Connect： 长期稳定连接本地数据中心和 AWS、低延迟、私有专线


### Question #7

Topic 1

A company has an application that ingests incoming messages（有一个接受传入信息的应用）. Dozens of other applications and microservices then quickly consume these

messages（快速分发给多个应用和微服务进行消费）. The number of messages varies drastically（突然变化得很快） and sometimes increases suddenly to 100,000 each second. The company wants to

decouple（解耦） the solution and increase scalability.

Which solution meets these requirements?

- 考点： 消息队列/发布订阅架构，考察队AWS 消息中间件的选型。
  - 解耦：生产者和消费这之间互不感知，独立扩展
  - Fan-out模式：一台哦消息需要同时推送给多个消费方
  - 高吞吐量处理：应对突发流量
  - 涉及服务：SNS，SQS，Kinesis，EC2 AutoScaling，Kinesis（动力） Data Analysis

A. Persist the messages to Amazon Kinesis Data Analytics. Configure the consumer applications to read and process the messages.

- Kinesis Data Analytics 用来对流数据做实时 SQL 分析的计算层（比如实时聚合，窗口计算）
- Kinesis Data Stream 才是用来做消息传输服务

B. Deploy the ingestion application on Amazon EC2 instances in an Auto Scaling group to scale the number of EC2 instances based on

CPU metrics.

- AS 基于 CPU 扩容有明显滞后性：流量突发增加到10万/秒，CPU指标触发 ➡️ 启动新实例 ➡️ 预热，这个过程需要数分钟，无法应对“突发”的流量暴增。
- 同时这个选项没有解耦。

C. Write the messages to Amazon Kinesis Data Streams with a single shard. Use an AWS Lambda function to preprocess messages and

store them in Amazon DynamoDB. Configure the consumer applications to read from DynamoDB to process the messages.

- 使用单个 shard（碎片） 是致命缺陷，Kinesis Data Streams with a single shard 的吞吐上限是：
  - 写入：1 MB/s 或 1,000 条/秒 （题目是100,000条/秒）
  - 读取：2 MB/s 

D. Publish the messages to an Amazon Simple Notification Service (Amazon SNS) topic with multiple Amazon Simple Queue Service

(Amazon SQS) subscriptions. Configure the consumer applications to process the messages from the queues.

- SNS + SQS
- 解耦：生产者只发布到 SNS Topic，完全不知道有多少消费者，消费者从各自SQS独立拉取
- Fan-out: SNS将一条消息Fan-out（扇出）到所有订阅的SQS队列，每个消费者应用拿到独立的副本
- 高吞吐：SQS 标准队列吞吐量近乎无限，SNS 也支持极高的TPS
- 数十个消费方：每个微服务订阅独立SQS队列，互不干扰
- 可扩展性：SQS队列深度可触发AS，消费侧可按需扩容

Correct Answer: D 

架构示意(这个模式叫做 SNS Fan-out Pattern，是 AWS 官方推荐的标准解耦架构。)：
```text
生产者 → SNS Topic → SQS Queue A → Consumer App 1
                   → SQS Queue B → Consumer App 2
                   → SQS Queue C → Consumer App 3
                   → ...（数十个）
```

核心知识点总结：
SNS     → 发布/订阅，推送模型，适合 Fan-out
SQS     → 队列，拉取模型，适合解耦 + 削峰填谷
SNS+SQS → Fan-out Pattern，生产者与多消费者完全解耦

Kinesis Data Streams  → 流式传输，有序，适合日志/事件流分析
Kinesis Data Analytics → 流数据实时 SQL 分析，≠ 消息队列

扩展：
1. SNS vs Kinesis
   - Kinesis：适合有序处理，需要回放（Reply），日志/事件流，单一消费者群
   - SNS + SQS： Fanout到异构多消费者，每个消费者需要独立副本，无需保证严格顺序

2. SQS的更多优势（真实系统设计中非常关键）
   - SQS可以解耦
   - 可见性超时（Visibility Timeout）: 防止消息被重复处理
   - 死信队列（DLQ）：处理失败的消息不丢失
   - 延迟队列：控制消费时机



### Question #8

Topic 1

A company is migrating a distributed application to AWS. The application serves variable workloads. The legacy（遗留的） platform consists of a primary

server that coordinates jobs across multiple compute nodes. The company wants to modernize the application with a solution that maximizes

resiliency(弹性) and scalability.

How should a solutions architect design the architecture to meet these requirements?

- 公司需要将一个分布式应用迁移到AWS，原架构中有一个 a primary server负责协调任务，并把多个任务分配个多个compute nodes。
- 需求：maximizes resiliency(弹性) and scalability，处理波动的工作负载
- 核心考点： SQS 解耦任务队列 + AC动态扩展 Worker，考察队分布式任务处理架构现代化的选型能力

A. Configure an Amazon Simple Queue Service (Amazon SQS) queue as a destination for the jobs. Implement the compute nodes with

Amazon EC2 instances that are managed in an Auto Scaling group. Configure EC2 Auto Scaling to use scheduled scaling.

- ❌ 在使用scheduled scaling，这个定时执行，不适合 variable workloads.

B. Configure an Amazon Simple Queue Service (Amazon SQS) queue as a destination for the jobs. Implement the compute nodes with

Amazon EC2 instances that are managed in an Auto Scaling group. Configure EC2 Auto Scaling based on the size of the queue.

- 解耦：使用SQS将任务生产者和compute nodes解耦，不再依赖 primary server直接协调
- 高弹性：SQS托管消息队列，可以缓冲突发任务
- 可扩展：使用 EC2 AS 可以根据队列积压数量增加或者减少 compute nodes
- 根据 size of SQS，也就是待处理任务的backlog 动态扩缩容
- 降低单点故障：使用SQS代替了原来的primary server ，减少中心协调节点带来的瓶颈和故障风险

架构示意图：
```text
Job Producer → Amazon SQS Queue → EC2 Auto Scaling Group Workers → Process Jobs
```

C. Implement the primary server and the compute nodes with Amazon EC2 instances that are managed in an Auto Scaling group.

Configure AWS CloudTrail as a destination for the jobs. Configure EC2 Auto Scaling based on the load on the primary server.

- CloudTrail 用来记录AWS API调用和账户活动，用于审计合规和安全分析

D. Implement the primary server and the compute nodes with Amazon EC2 instances that are managed in an Auto Scaling group.

Configure Amazon EventBridge (Amazon CloudWatch Events) as a destination for the jobs. Configure EC2 Auto Scaling based on the load

on the compute nodes.

- EventBridge 适用于事件驱动架构中的事件路由、规则匹配、SaaS/AWS服务事件集成

Correct Answer: B 

核心知识点总结：
```text
Amazon SQS → 托管消息队列 / 用于解耦生产者和消费者，缓冲突发任务，平滑工作负载

EC2 Auto Scaling Group → 自动管理一组 EC2 worker / 根据指标增加或减少计算节点

SQS Queue Size / ApproximateNumberOfMessagesVisible → 常用于 worker 扩缩容判断 / 队列越长，说明待处理任务越多

Scheduled Scaling → 适合有明确时间规律的负载 / 不适合不可预测的 variable workloads

Amazon EventBridge → 事件总线和事件路由 / 适合事件驱动集成，不是典型 job backlog 队列

AWS CloudTrail → 审计 AWS API 调用 / 不是消息队列，也不能作为任务处理 destination

SQS + Auto Scaling Group → 典型分布式任务处理架构 / Job Producer 写入队列，Worker Fleet 从队列拉取任务处理

考试判断规则 → 看到 variable workloads + decouple + compute workers + scalability，优先考虑 SQS + Auto Scaling based on queue depth

记忆口诀 → “任务波动要缓冲，Worker 扩缩看队列”
```




### Question #9

Topic 1

A company is running an SMB file server in its data center. The file server stores large files that are accessed frequently for the first few days

after the files are created. After 7 days the files are rarely accessed.

The total data size is increasing and is close to the company's total storage capacity. A solutions architect must increase the company's

available storage space without losing low-latency access to the most recently accessed files. The solutions architect must also provide file

lifecycle management to avoid future storage issues.

Which solution will meet these requirements?

- 场景：公司在本地数据中心运行SMB文件服务器，创建前几天访问频繁，7天后少有访问。
- 当前问题：本地存储容量接近上限，需要扩展可用存储空间
- 需求：扩展存储空间，保持最近访问的文件低延迟访问，提供生命周期管理
- 考点：考察对混合云文件的存储扩展，本地缓存和对象生命周期管理能力（S3 File Gateway 和 S3 Lifecycle）

A. Use AWS DataSync to copy data that is older than 7 days from the SMB file server to AWS.

- DataSync 提供的是数据迁移和同步的服务

B. Create an Amazon S3 File Gateway to extend the company's storage space. Create an S3 Lifecycle policy to transition the data to S3 Glacier Deep Archive after 7 days.

- S3 File Gateway 将文件以 SMB/NFS 方式暴露给本地用户或应用，后段数据存储在 S3 中，避免继续依赖本地文件服务器容量；同时在本地保留缓存，保持最近文件延迟访问；支持SMB文件共享；

- S3 Glacier Deep Archive 适合极低频访问，可接受较长恢复时间的数据
- 但如果强调 “7天后仍需要快速读取”，选择S3 Standard-IA、S3 One Zone-IA 或 S3 Glacier Instant Retrieval。

C. Create an Amazon FSx for Windows File Server file system to extend the company's storage space.
