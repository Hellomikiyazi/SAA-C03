## 1. S3、对象存储与生命周期（15 次）

涉及题号：#1, #2, #3, #4, #6, #9, #12, #18, #21, #22, #23, #32, #36, #38, #40

关联内容：

- `S3 Transfer Acceleration`：全球站点高速上传到单一 S3 bucket，低运维，适合 #1 的跨洲数据聚合。
- `S3 + Athena`：日志已经在 S3 且 JSON 格式，临时 SQL 查询选 Athena，避免搬到 Redshift/EMR，见 #2。
- `S3 bucket policy + aws:PrincipalOrgID`：限制只有同一个 AWS Organization 内的账号主体可访问，见 #3。
- `S3 Gateway VPC Endpoint`：VPC 内 EC2 私网访问 S3，不走 Internet/NAT/IGW，见 #4。
- `Snowball Edge -> S3`：几十 TB 级一次性迁移且尽量不用网络带宽，见 #6。
- `S3 File Gateway + S3 Lifecycle`：本地 SMB/NFS 文件场景扩展容量，保留热数据本地低延迟缓存，冷数据转低成本层，见 #9。
- `CloudFront + S3 origin`：静态网站或静态资源全球低延迟访问，见 #12、#38。
- `S3 Event Notification -> SQS -> Lambda`：上传对象后自动、持久、无状态处理，见 #18。
- `S3 static website hosting`：纯 HTML/CSS/JS/图片站点最省钱方案，见 #32。
- `S3 Intelligent-Tiering`：访问模式不可预测，同时要求跨 AZ 韧性，见 #22。
- `S3 Lifecycle -> Glacier Deep Archive/Glacier`：明确 1 个月后不访问且长期保留，转 Deep Archive；14 天后归档分析数据，见 #23、#40。
- `SSE-KMS / multi-Region KMS key`：跨区域 S3 加密题要注意“同一个 KMS key”和“每个 Region 都存 key”的 wording，见 #36。

判断关键词：

- `global upload`, `single S3 bucket`, `as quickly as possible` -> S3 Transfer Acceleration。
- `query JSON/CSV/Parquet in S3`, `on-demand`, `least overhead` -> Athena。
- `private subnet access S3`, `without internet` -> Gateway VPC Endpoint。
- `unpredictable access pattern` -> S3 Intelligent-Tiering。
- `static website`, `cost-effective` -> S3 hosting；全球低延迟再加 CloudFront。

## 2. 安全、IAM、凭证、合规与治理（13 次）

涉及题号：#3, #11, #13, #15, #17, #26, #27, #28, #31, #34, #35, #36, #37

关联内容：

- `aws:PrincipalOrgID`：S3 bucket policy 限定组织内账号访问，见 #3。
- `AWS Secrets Manager`：数据库用户名/密码集中管理和自动轮换，尤其 Aurora/RDS，见 #11、#13。
- `Secrets Manager multi-Region secret replication`：跨 Region 数据库凭证轮换，见 #13。
- `IAM Role for EC2 / Instance Profile`：EC2 访问 S3 等 AWS 服务，不使用 IAM User 长期密钥，见 #17。
- `AWS Config`：检查 S3 bucket 未授权配置变更、资源标签合规、资源配置历史，见 #26、#31、#34。
- `AWS CloudTrail`：记录 AWS API 调用历史，与 Config 配合满足审计要求，见 #34。
- `AWS Network Firewall`：VPC 进出流量 inspection/filtering，见 #15。
- `AWS Shield Advanced`：面向 ELB/CloudFront/Route 53 等资源的大规模 DDoS 防护，见 #35。
- `AWS Systems Manager Session Manager`：原生、可重复、低运维的 EC2 远程管理方式，见 #37。
- `AWS IAM Identity Center / AWS SSO + AWS Managed Microsoft AD trust`：组织多账号 SSO，并继续使用本地自管 AD 用户组，见 #28。

判断关键词：

- `database credentials`, `automatic rotation` -> Secrets Manager。
- `configuration changes`, `compliance`, `required tags` -> AWS Config。
- `API calls`, `audit history` -> CloudTrail。
- `EC2 access S3`, `least privilege`, `no long-term keys` -> IAM Role。
- `remote administer EC2`, `native AWS`, `least overhead` -> Systems Manager Session Manager。

## 3. 数据库、分析与 BI（11 次）

涉及题号：#2, #11, #13, #14, #16, #21, #24, #30, #33, #39, #40

关联内容：

- `Athena`：直接查询 S3 上的 JSON 日志，适合简单临时 SQL，见 #2。
- `Aurora + Aurora Replicas + Auto Scaling`：MySQL 读多写少、读流量不可预测、高可用，见 #14。
- `QuickSight`：连接 S3、RDS PostgreSQL 等多数据源做可视化 dashboard；用 QuickSight users/groups 做访问控制，见 #16。
- `DynamoDB`：高并发、毫秒级低延迟、Serverless Web 后端数据存储，见 #21。
- `Kinesis Data Streams + Lambda + DynamoDB`：金融交易近实时流处理，脱敏后写入文档/键值数据库，同时让多个内部应用消费流，见 #33。
- `Kinesis Data Firehose -> S3 + Lifecycle`：海量告警摄取、低运维、后续分析与归档，见 #40。
- `RDS MySQL 测试环境成本`：每月只跑 48 小时、要求保留实例规格，通常快照后删除实例，需要时恢复，见 #30。
- `RDS storage performance`：问题明确是存储 I/O 性能时，通常切到 Provisioned IOPS SSD，见 #39 的疑似订正。
- `Cost Explorer`：按实例类型等维度做账单根因分析，见 #24 的疑似订正。

判断关键词：

- `visualization`, `dashboard`, `users/groups` -> QuickSight。
- `read requests more than writes`, `unpredictable read workloads` -> Aurora Replicas Auto Scaling。
- `millions requests`, `millisecond latency`, `least overhead` -> DynamoDB。
- `near-real-time`, `millions transactions`, `several applications consume` -> Kinesis Data Streams。
- `ingest and store`, `no infrastructure`, `archive older than N days` -> Kinesis Data Firehose + S3 Lifecycle。

## 4. 计算、扩展与 Web 架构（9 次）

涉及题号：#5, #8, #14, #21, #25, #29, #30, #32, #37

关联内容：

- `EFS`：多台 EC2 跨 AZ 共享用户上传文件，替代各自独立 EBS，见 #5。
- `SQS + EC2 Auto Scaling by queue depth`：分布式任务 Worker 现代化，替代 primary server 协调，见 #8。
- `ALB + EC2 Auto Scaling`：应用层扩展常见基础架构，但不解决共享数据或数据库读扩展问题，见 #5、#14。
- `Serverless Web`：S3 静态内容 + CloudFront + API Gateway + Lambda + DynamoDB，适合高峰每小时百万请求、毫秒级延迟、低运维，见 #21。
- `Lambda + SQS`：API Gateway 高并发写入 Aurora 时，拆分接收函数和加载函数，中间用 SQS 缓冲，见 #25。
- `Systems Manager Session Manager`：远程管理 EC2，不需要堡垒机和 SSH key 分发，见 #37。
- `Static website on S3`：纯静态站点最省钱，不需要 Fargate/EC2/ALB+Lambda，见 #32。

判断关键词：

- `multiple EC2 need same files` -> EFS。
- `variable workloads`, `compute nodes`, `jobs` -> SQS + ASG based on queue size。
- `least operational overhead`, `millions requests` -> Serverless managed services。
- `Lambda quota too high`, `database loading burst` -> SQS 缓冲削峰。

## 5. 消息队列、事件驱动与流处理（8 次）

涉及题号：#7, #8, #10, #18, #21, #25, #33, #40

关联内容：

- `SNS + SQS Fan-out`：一条消息需要多个应用/微服务快速消费，每个消费者独立队列，见 #7。
- `SQS queue depth scaling`：任务波动时，Worker 根据 backlog 扩缩容，见 #8。
- `SQS FIFO`：订单必须按接收顺序处理，见 #10。
- `S3 Event Notification + SQS + Lambda`：图片上传后自动压缩处理，SQS 提供持久缓冲，见 #18。
- `SQS between Lambda functions`：高吞吐摄取和数据库写入解耦，见 #25。
- `Kinesis Data Streams`：近实时、大规模事务流，多消费者分析与处理，见 #33。
- `Kinesis Data Firehose`：托管摄取并落 S3，适合不想管理消费者基础设施的日志/告警归档，见 #40。

判断关键词：

- `fan-out`, `dozens of applications`, `multiple consumers` -> SNS topic + SQS subscriptions。
- `process in order` -> SQS FIFO。
- `durable, stateless, automatically process` -> SQS + Lambda。
- `stream`, `near-real-time`, `multiple internal applications consume` -> Kinesis Data Streams。
- `delivery to S3`, `no infrastructure` -> Kinesis Data Firehose。

## 6. 网络、边缘加速与流量检查（7 次）

涉及题号：#1, #4, #12, #15, #19, #29, #38

关联内容：

- `S3 Transfer Acceleration`：利用 AWS edge location 加速全球上传，见 #1。
- `Gateway VPC Endpoint for S3/DynamoDB`：私网访问 S3，见 #4。
- `CloudFront multi-origin`：S3 静态内容 + ALB 动态内容统一加速，Route 53 alias 指向 CloudFront，见 #12。
- `CloudFront for S3 static website`：全球用户访问静态站点降低延迟，见 #38。
- `AWS Network Firewall`：托管 VPC 流量检查和过滤，见 #15。
- `Gateway Load Balancer`：第三方虚拟防火墙/IDS/IPS 设备透明接入流量路径，见 #19。
- `Global Accelerator + NLB`：UDP/VoIP、多 Region、最低延迟和自动故障转移，见 #29 的疑似订正。

判断关键词：

- `static and dynamic content`, `global users`, `Route 53 domain` -> CloudFront with S3 and ALB origins。
- `third-party appliance`, `accept IP packets`, `inspect all traffic` -> Gateway Load Balancer。
- `traffic inspection and filtering in VPC` -> Network Firewall。
- `UDP`, `multi-Region`, `lowest latency`, `automated failover` -> Global Accelerator + NLB。

## 7. 成本优化与账单分析（5 次）

涉及题号：#23, #24, #30, #31, #40

关联内容：

- `S3 Lifecycle`：频繁访问 1 个月后不再访问并永久保留，转 Glacier Deep Archive 最省钱，见 #23。
- `Cost Explorer granular filtering`：按 EC2 instance type 等维度做 2 个月成本对比和根因分析，见 #24 的疑似订正。
- `RDS snapshot + terminate + restore`：每月仅使用 48 小时且不降低规格，删除实例只保留快照更省钱，见 #30。
- `AWS Config required-tags`：低运维发现未正确打标签资源，帮助成本分摊，见 #31。
- `Kinesis Firehose + S3 Lifecycle`：摄取后 14 天热数据，之后归档，见 #40。
