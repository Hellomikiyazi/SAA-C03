### 【考点 1】S3、CloudFront 与对象存储治理

**出现频率：21题（最高频）**  
**涉及题号：106, 109, 110, 116, 118, 126, 131, 132, 134, 147, 153, 154, 155, 158, 160, 165, 166, 172, 173, 183, 189**

#### 1.1 S3 存储类与生命周期

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q118 | 900 TB 文本文档，高并发访问，低成本 | **Amazon S3** | 海量对象存储、自动扩展、成本最低 |
| Q126 | 最近 2 年需立即访问，之后保留 25 年 | **S3 Lifecycle：2 年后转 Glacier Deep Archive** | 满足近期可用性与长期最低成本 |
| Q147 | 日志保留 10 年，近 1 个月常查 | **S3 + Lifecycle：1 个月后转 Glacier Deep Archive** | 长期归档成本最低 |
| Q153 | 铃声文件 90 天后低频访问但需可用 | **S3 Standard -> Standard-IA 生命周期** | 低频但仍需快速访问 |
| Q160 | 300 MB JSON 灾备，30 天内毫秒级访问 | **S3 Standard** | Glacier 不满足毫秒级访问 |

**核心判断规则：**
- 需要毫秒级访问：S3 Standard / Standard-IA，不能选 Glacier。
- 明确冷热时间线：优先 S3 Lifecycle。
- 长期极少访问且允许恢复延迟：Glacier Deep Archive。
- 海量静态对象、无需文件系统语义：优先 S3。

#### 1.2 S3 对象防篡改与合规保留

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q109 | 新对象不确定保留多久，特定用户可解除 | **S3 Object Lock + Legal Hold** | Legal Hold 无需固定保留期 |
| Q154 | 医疗文件至少 1 年不可修改/删除 | **Object Lock Compliance Mode + 365 天保留期** | Compliance 模式无法被普通权限绕过 |
| Q189 | 合同 5 年不可覆盖删除，需加密与自动轮换 | **Object Lock Compliance + SSE-KMS 自动轮换** ⚠️ | 社区高票为 BD；源文件答案 CE 疑似错误 |

**核心判断规则：**
- 固定保留期、任何人不得删除：Object Lock Compliance Mode。
- 不确定保留期、需要人工解除：Legal Hold。
- Governance Mode 可被有权限用户绕过；Compliance Mode 不能被绕过。

#### 1.3 S3 加密、复制与私有访问

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q106 | S3 机密数据，记录密钥使用，年度轮换 | **SSE-KMS + 自动轮换** | KMS 有 CloudTrail 审计与自动轮换 |
| Q134 | S3 数据需加密、跨 Region 复制、SQL 查询 | **S3 CRR + SSE-KMS Multi-Region Keys + Athena** ⚠️ | 社区在 SSE-S3 与 SSE-KMS 间有分歧 |
| Q131 | CloudFront 分发 S3 文件，禁止直连 S3 URL | **CloudFront OAI + S3 Bucket Policy** | 只允许 CloudFront 读取源桶 |
| Q165 | CloudFront + S3 静态站点，所有流量经 WAF 检查 | **CloudFront OAI + 关联 AWS WAF** | 阻止绕过 CloudFront 直连 S3 |

**核心判断规则：**
- S3 加密需审计密钥使用：SSE-KMS。
- S3 不允许被绕过访问：OAI/OAC + Bucket Policy。
- WAF 检查 S3 静态站点流量：WAF 关联 CloudFront，S3 源站限制 CloudFront 访问。

#### 1.4 CloudFront 内容分发与流媒体


| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q116 | 静态官网，少量更新，高扩展高安全 | **S3 Static Website + CloudFront HTTPS** | 最低运维，无服务器静态托管 |
| Q132 | 全球下载历史报告，最快响应，少基础设施 | **CloudFront + S3** | 边缘缓存静态文件 |
| Q155 | 全球可靠快速访问 S3 私密媒体 | **CloudFront 连接 S3 边缘缓存** | 降低延迟与源站压力 |
| Q158 | 全球实时与点播视频 | **CloudFront** | 同时优化直播与点播分发 |
| Q166 | 全球静态 HTML 每日报告，百万访问 | **CloudFront + S3 Origin** | 静态内容全球缓存 |
| Q173 | 全球大量用户访问同一批 S3 图片视频 | **CloudFront Web Distribution** | 最低成本降低源站负载 |
| Q183 | 动态订购网站，少维护，高可用并快速扩展读写 | **S3 + CloudFront + API Gateway/Lambda + DynamoDB On-Demand** | 静态内容边缘分发，动态逻辑无服务器 |
| Q172 | HTTPS 外还需保护敏感字段贯穿应用栈 | **CloudFront Field-Level Encryption** ⚠️ | 社区 77% 选 C；源文件答案 A 疑似错误 |

**核心判断规则：**
- 静态内容全球访问：S3 + CloudFront。
- 动静态内容都要低延迟：CloudFront 可把 ALB/API 作为 Origin。
- 保护表单敏感字段到特定后端应用：Field-Level Encryption，不是 Signed URL/Cookie。

### 【考点 2】网络架构、边缘加速与 Web 防护

**出现频率：17题**  
**涉及题号：101, 104, 115, 119, 120, 123, 125, 135, 141, 142, 159, 165, 169, 170, 176, 180, 184**

#### 2.1 NAT、VPC Endpoint 与私有连接

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q101 | 3 AZ 私有子网访问互联网更新软件 | **每个 AZ 一个 NAT Gateway + 私有路由表指向本 AZ NAT** | 高可用并避免跨 AZ NAT 依赖 |
| Q115 | EC2 只需私有访问 S3，不走互联网 | **私有子网 + S3 VPC Endpoint** | S3 流量留在 AWS 网络内 |
| Q125 | 两层电商应用，EC2/RDS 不暴露公网但 EC2 需访问第三方支付 | **公有 ALB + 私有 EC2/RDS + 多 AZ NAT Gateway** ⚠️ | 题目选项标签重复；核心架构是公有入口、私有计算与数据库 |
| Q176 | 私有 EC2 访问 DynamoDB，不离开 AWS 网络 | **DynamoDB VPC Endpoint** ⚠️ | 社区 100% 选 A；源文件答案 D 明显错误 |
| Q135 | 只从本方 VPC 私有访问外部提供商目标服务 | **AWS PrivateLink / VPC Endpoint Service** | 单向、私有、只暴露目标服务 |
| Q184 | Lambda 访问本地数据中心私有数据库 | **Lambda 配置到 VPC + 路由经 Direct Connect** ⚠️ | 社区偏 A；源文件答案 C 强调路由表 |

**核心判断规则：**
- 私有子网访问互联网：NAT Gateway 放公有子网。
- S3/DynamoDB 私网访问：Gateway VPC Endpoint。
- 访问第三方 VPC 中特定服务：PrivateLink。
- Lambda 访问 VPC/本地私网资源：Lambda 必须配置 VPC 子网和安全组。

#### 2.2 边缘加速与全球入口

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q120 | 多 Region NLB 承载 DNS 服务，需性能与可用性 | **Global Accelerator + endpoint groups + NLB** ⚠️ | 社区 75% 选 B；源文件答案 A 疑似错误 |
| Q141 | 全球门户，HTTPS，动静态混合内容最低延迟 | **CloudFront + ALB Origin** ⚠️ | 社区 70% 选 A；源文件答案 B 为多 Region Route 53 |
| Q142 | UDP 游戏，最近边缘、低延迟、固定入口 IP | **Global Accelerator + NLB + EC2 Auto Scaling** | Global Accelerator 支持静态 Anycast IP 与 UDP |
| Q123 | EC2 上 SSL 开销过高 | **ACM 导入证书 + ALB HTTPS Listener 终止 TLS** | 把 TLS 卸载到负载均衡层 |

**核心判断规则：**
- TCP/UDP、固定 Anycast IP、全球加速：Global Accelerator。
- HTTP/HTTPS 内容分发与缓存：CloudFront。
- TLS 卸载：ALB/NLB + ACM，避免 EC2 承担证书运算。

#### 2.3 WAF、Shield 与 DDoS/Web 攻击防护

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q104 | 大规模 DDoS，网站不可停机 | **AWS Shield Advanced + CloudFront** | Shield Advanced 防 DDoS，CloudFront 吸收边缘流量 |
| Q119 | 多账户、多 Region API Gateway 防 SQLi/XSS | **AWS Firewall Manager 集中配置 WAF** ⚠️ | 多账户最少管理应选 Firewall Manager；源文件答案 A |
| Q159 | API Gateway + Lambda 遭 botnet 欺诈请求 | **AWS WAF + API 访问控制** ⚠️ | 社区偏 WAF + API Key；源文件选 WAF + Private API |
| Q169 | ALB 公网应用降低 DDoS 风险 | **AWS Shield Advanced** | 针对复杂 DDoS 的增强防护 |
| Q170 | ALB 应用只允许特定国家访问 | **AWS WAF Geo Match** | 安全组/NACL 无国家维度 |
| Q180 | API Gateway + NLB，防 SQLi 与高级 DDoS | **WAF 保护 API Gateway + Shield Advanced 保护 NLB** | WAF 处理 L7，Shield 处理 DDoS |
| Q165 | CloudFront + S3 所有流量经 WAF | **WAF 关联 CloudFront + OAI 限制 S3** | 防止绕过 WAF 访问源站 |

**核心判断规则：**
- SQL 注入/XSS/地理限制：AWS WAF。
- 多账户集中 WAF 策略：AWS Firewall Manager。
- DDoS 增强防护：AWS Shield Advanced。
- GuardDuty 是威胁检测，不是流量阻断方案。

### 【考点 3】容器、计算扩展与成本优化

#### 3.1 容器托管与迁移

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q112 | 本地容器 Web 应用迁 AWS，少改代码少运维 | **ECS on Fargate + Service Auto Scaling + ALB** | 无需管理服务器 |
| Q129 | 本地容器 + PostgreSQL，运维和容量规划负担大 | **ECS Fargate + Aurora** | 应用和数据库均托管化 |
| Q143 | 单体应用拆成小应用，不大改前后端 | **ECS + ALB** | 容器化拆分，团队独立管理 |
| Q163 | 容器规模化部署，最小运维 | **ECR + ECS Fargate + Target Tracking** ⚠️ | 社区 100% 选 A；源文件答案 C 疑似错误 |
| Q187 | 电商：负载均衡前端、容器应用、关系库，高可用少人工 | **ECS Fargate + RDS Multi-AZ** | 托管容器 + 托管数据库 HA |

- 运行容器且少运维：ECS Fargate。
- 已有 Kubernetes 部署方式且不能改：EKS。
- ECS 应用访问 AWS 服务：Task Role。
- 关系数据库高可用：RDS Multi-AZ。

#### 3.2 Auto Scaling 与高可用

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q130 | ASG 中 EC2 CPU 保持约 40% | **Target Tracking Scaling Policy** | 按目标指标自动调节容量 |
| Q145 | 单 EC2 上 PHP + MySQL 性能下降 | **数据库迁 Aurora，Web 层 ASG + ALB** | 分层扩展，Web 层横向伸缩 |
| Q174 | 6 台前端单 AZ，需要高可用 | **ASG 跨两个 AZ 部署** | 多 AZ 抗单 AZ 故障 |
| Q190 | Java/PHP 应用迁 AWS，频繁测试新特性 | **Elastic Beanstalk + URL Swapping** ⚠️ | 社区 90% 选 B；源文件答案 D 疑似错误 |

**核心判断规则：**
- 维持某个平均指标：Target Tracking。
- 高可用基础：跨多个 AZ。
- Web 层扩展：ALB + Auto Scaling。
- Elastic Beanstalk 适合托管 Web 应用和蓝绿/URL swap。

#### 3.3 计算成本优化

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q124 | 无状态批处理，可中断，60 分钟完成 | **EC2 Spot Instances** | 可中断工作负载最低成本 |
| Q128 | 无状态容器，可容忍底层中断 | **Spot 容量运行容器** ⚠️ | 社区偏 EKS Managed Node Group Spot；源文件选 EC2 ASG Spot |
| Q140 | EC2 不规律可中断；Fargate/Lambda 可预测一年 | **EC2 用 Spot；Fargate/Lambda 用 Compute Savings Plan** | 不同计算形态分别优化 |
| Q146 | 工作日高峰、夜间基线、周末低 | **RI 覆盖基线 + Spot 覆盖额外容量** | 基线稳定用 RI，弹性部分用 Spot |
| Q167 | SQS 消费端持续处理、流量不可预测 | **RI 基线 + Spot 额外容量** ⚠️ | 队列可缓冲 Spot 中断风险 |

**核心判断规则：**
- 可中断、无状态、队列缓冲：Spot。
- 长期稳定基线：Reserved Instances 或 Savings Plans。
- Fargate/Lambda/EC2 混合且可预测：Compute Savings Plan 覆盖面更广。

### 【考点 4】数据库高可用、读写分离与迁移

**出现频率：19题**  
**涉及题号：111, 121, 129, 133, 136, 138, 144, 152, 157, 175, 177, 178, 182, 183, 191, 193, 194, 196, 197**

#### 4.1 RDS/Aurora 高可用与连接管理

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q111 | ActiveMQ + 消费者 + MySQL on EC2 高可用低运维 | **Amazon MQ active/standby + ASG + RDS MySQL Multi-AZ** | 全链路托管化、多 AZ |
| Q138 | RabbitMQ + 应用 + PostgreSQL 单 AZ 改造 | **Amazon MQ RabbitMQ active/standby + ASG + RDS PostgreSQL Multi-AZ** | 队列与数据库托管 HA |
| Q175 | Lambda 突增连接压垮 Aurora PostgreSQL | **RDS Proxy + Lambda 改连 Proxy Endpoint** | 连接池复用，最少应用改动 |
| Q182 | MySQL 上云，最小数据丢失，事务至少两节点 | **RDS MySQL Multi-AZ** | 同步复制到备用 AZ |
| Q194 | 必须用 EC2 承载数据库且自动故障转移 | **跨 AZ EC2 数据库集群 + 复制** ⚠️ | 社区偏 A；源文件答案 C 跨 Region 不一定自动 |

**核心判断规则：**
- RDS Multi-AZ：高可用/故障转移，不用于读扩展。
- Lambda 连接 RDS/Aurora 过多：RDS Proxy。
- 自建数据库必须 HA：跨 AZ 集群 + 数据库层复制/故障转移。

#### 4.2 读扩展与报表隔离

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q144 | Aurora 月报导致 ReadIOPS/CPU 峰值 | **Aurora Replica 承载报表查询** | 报表读流量与主库隔离 |
| Q191 | RDS MySQL 报表查询导致订单超时 | **创建 Read Replica，报表查只读副本** ⚠️ | 社区 100% 选 A；源文件答案 B 表述错误 |
| Q193 | 批处理对多个 RDS 高读压力，需减少数据库读取 | **ElastiCache for Redis 或 Read Replica** ⚠️ | 社区略偏 Redis；源文件答案 A |

**核心判断规则：**
- 报表查询影响 OLTP：Read Replica/Aurora Replica。
- 读请求重复且可缓存：ElastiCache。
- Multi-AZ Standby 不承担读请求。

#### 4.3 数据库迁移、备份与成本

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q121 | 未加密 RDS Multi-AZ 与快照以后都要加密 | **复制最新快照并加密，恢复为新 DB** | RDS 不能原地启用加密 |
| Q133 | Oracle 迁 AWS，升级、DR、低运维且需 OS 访问 | **RDS Custom for Oracle + 跨 Region Read Replica** ⚠️ | 社区偏 C；源文件答案 D 不满足 OS 访问 |
| Q136 | 本地 PostgreSQL 在线迁 Aurora PostgreSQL 并保持同步 | **AWS DMS 复制实例 + ongoing replication task** ⚠️ | 社区 AC；源文件 CD 含 SCT |
| Q152 | RDS MySQL 每天只用 12 小时，降成本 | **EventBridge 定时触发 Lambda start/stop DB** | 自动开停机 |
| Q157 | Aurora 数据保留 5 年后删除，审计日志永久保留 | **AWS Backup 保留 5 年 + CloudWatch Logs Export** ⚠️ | 社区 DE；源文件 BE 疑似错误 |
| Q178 | EC2 与 RDS 跨 Region 备份，最低运维 | **AWS Backup 跨 Region copy** | 统一管理 EC2/RDS 备份 |
| Q197 | .NET + Oracle 迁 AWS，少改动，高可用 | **Elastic Beanstalk .NET Multi-AZ + Oracle on RDS Multi-AZ** ⚠️ | 社区 BE；源文件 BD 中 DynamoDB 不合理 |

**核心判断规则：**
- RDS 加密：加密快照副本后恢复。
- 在线异构/同构迁移：DMS 复制实例 + replication task。
- 需要 OS 访问的托管 Oracle：RDS Custom。
- 统一跨服务备份：AWS Backup。

#### 4.4 DynamoDB 性能与生命周期

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q177 | DynamoDB 读密集且延迟高，不能增加运维 | **DynamoDB Accelerator (DAX)** | 托管缓存层，应用改动小 |
| Q183 | 动态订购网站，读写容量需快速扩展 | **DynamoDB On-Demand Capacity** | 无需容量规划，自动适应突增 |
| Q196 | 表持续增长，只需最近 30 天数据 | **应用写入过期时间属性 + DynamoDB TTL** | 最低开发与运维成本自动过期 |

**核心判断规则：**
- DynamoDB 读密集低延迟：DAX。
- 流量不可预测、读写需快速扩展：On-Demand。

## 四、快速记忆清单

- **S3 生命周期**：明确冷热边界用 Lifecycle；立即访问用 Standard/IA；最低长期归档用 Glacier Deep Archive。
- **S3 合规不可删**：固定期限选 Object Lock Compliance；不确定期限选 Legal Hold。
- **CloudFront**：静态、动态、视频分发均可加速；S3 源站要配 OAI/OAC 防直连。
- **Global Accelerator**：TCP/UDP、固定 Anycast IP、NLB、多 Region 入口。
- **WAF vs Shield**：WAF 防 SQLi/XSS/Geo；Shield Advanced 防高级 DDoS。
- **VPC Endpoint**：S3/DynamoDB 用 Gateway Endpoint；第三方服务私有访问用 PrivateLink。
- **ECS/Fargate**：容器低运维首选；EKS 用于不能改变 Kubernetes 部署模型。
- **RDS Multi-AZ**：高可用故障转移；Read Replica/Aurora Replica 才是读扩展。
- **Lambda + RDS**：连接数过多用 RDS Proxy。
- **SQS**：异步解耦、缓冲故障、最终处理；保序用 FIFO。
- **DataSync/Snowball**：在线自动迁移用 DataSync；几十 TB 且无带宽用 Snowball Edge。
- **AI/ML 托管服务**：Textract 做 OCR，Comprehend Medical 抽医学实体，Transcribe 做语音转文字，Pinpoint 做营销触达。
