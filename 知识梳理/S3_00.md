S3 全称是 Amazon Simple Storage Service，中文通常叫 Amazon S3 对象存储服务。它不是传统硬盘，也不是数据库，而是 AWS 上用来存放“对象数据”的托管存储服务，适合存图片、视频、日志、备份、静态网站文件、数据湖文件、归档数据等。AWS 官方定义里，S3 是一个对象存储服务，特点是高扩展性、高可用性、安全性和高性能，可用于数据湖、网站、移动应用、备份恢复、归档、IoT、大数据分析等场景。 

1. 先理解：S3 是“对象存储”

传统文件系统里，你会这样理解数据：

/Users/miki/photos/2026/child.jpg

它是目录 + 文件的结构。

S3 里不是这种真正的目录结构，而是：

Bucket: my-photo-bucket
Object Key: photos/2026/child.jpg
Object Data: 图片本体
Metadata: 类型、大小、加密方式、标签等

所以 S3 的核心单位是：

Bucket + Object + Key

Bucket 可以理解为一个顶层容器，类似“存储桶”。Object 是你真正存进去的数据，比如一张图片、一个 PDF、一个日志文件、一个视频。Key 是对象的名字或路径，例如 logs/2026/06/17/app.log。S3 看起来像文件夹，是因为 key 里面用了 /，但本质上它是对象 key，不是传统文件系统目录。

2. S3 和 EBS、EFS 的区别

S3 经常和 EBS、EFS 混淆。

S3 是对象存储。适合存图片、视频、日志、备份、静态网页、归档数据。应用通常通过 HTTP API、SDK、CLI 访问它。

EBS 是块存储。它像云上的硬盘，通常挂载到一台 EC2 实例上，用来安装系统、数据库、应用程序数据等。

EFS 是文件存储。它像共享文件服务器，可以被多台 EC2 同时挂载，适合多个服务器共享文件。

判断方法很简单：

要存对象、图片、日志、备份、静态资源 → S3
要给一台 EC2 挂硬盘 → EBS
多台 EC2 要共享文件系统 → EFS

在 SAA-C03 题目里，看到 object storage、static website、logs in S3、backup、archive、lifecycle、global upload，基本优先考虑 S3。你上传的考点文件里也把 “S3、对象存储、生命周期、迁移与静态站点” 列为高频考点，并强调看到对象存储、静态内容、全球上传、生命周期、低成本归档时，优先从 S3 体系选。 

3. S3 的基本结构

S3 的结构可以这样看：

AWS Account
└── S3 Bucket
    ├── Object: image/a.jpg
    ├── Object: logs/app.log
    ├── Object: backup/db.dump
    └── Object: website/index.html

一个 bucket 里面可以放很多 object。每个 object 都有一个 key。你访问对象时，本质上是通过 bucket 名称和 key 找到对象。

例如：

s3://my-bucket/images/cat.jpg

含义是：

Bucket = my-bucket
Key = images/cat.jpg

4. S3 为什么适合大规模存储？

S3 的设计目标不是“像本地磁盘一样挂载使用”，而是“海量对象可靠存储”。AWS 官方页面强调 S3 面向几乎所有使用场景，可存储大量数据，并具备扩展性、可用性、安全性和性能能力。 

所以它特别适合：

图片、视频、音频
日志文件
备份文件
静态网页
数据湖
机器学习训练数据
归档数据
导出文件
用户上传文件

但它不适合：

频繁随机写入的小块数据库文件
需要 POSIX 文件系统语义的应用
需要像硬盘一样直接 mount 的主存储
低延迟块级 I/O 场景

这类场景一般看 EBS、EFS、FSx 或数据库服务。

5. S3 的访问方式

S3 通常通过这些方式访问：

AWS Console
AWS CLI
AWS SDK
REST API
CloudFront
Athena
Lambda
EC2 IAM Role
VPC Endpoint

例如你可以用 AWS CLI 上传文件：

aws s3 cp ./index.html s3://my-bucket/index.html

也可以让程序通过 SDK 上传用户图片：

Web App → S3 Bucket

如果 EC2 在私有子网里访问 S3，不想经过 Internet、NAT Gateway 或 Internet Gateway，典型做法是用 S3 Gateway VPC Endpoint。你的考点文件里也把这个总结为：VPC 内 EC2 私网访问 S3，不走 Internet/NAT/IGW。 

6. S3 的权限控制

S3 默认不应该公开。它的权限通常由这些机制控制：

IAM Policy
Bucket Policy
S3 Block Public Access
Object Ownership
Access Point
ACL，旧机制，现代架构中尽量少用

AWS 官方安全最佳实践提到，S3 Object Ownership 默认采用 bucket owner enforced 设置，ACL 默认禁用，bucket owner 拥有对象并通过访问管理策略控制访问。 

常见判断：

EC2 访问 S3 → IAM Role / Instance Profile
限制某个 Bucket 只能被组织内账号访问 → Bucket Policy + aws:PrincipalOrgID
禁止公网误开放 → S3 Block Public Access
跨账号访问 → Bucket Policy / IAM Role

你的文件中也提到，aws:PrincipalOrgID 可用于限制只有同一个 AWS Organization 内的账号主体访问 S3 bucket。 

7. S3 的存储类型 Storage Class

S3 不是只有一种存储。每个对象都可以选择不同的 storage class。AWS 官方文档说明，默认对象存储在 S3 Standard，但 S3 提供多种存储类别，你可以根据访问模式、性能、可用性和成本需求选择。 

常见存储类型：

S3 Standard
S3 Intelligent-Tiering
S3 Standard-IA
S3 One Zone-IA
S3 Glacier Instant Retrieval
S3 Glacier Flexible Retrieval
S3 Glacier Deep Archive
S3 Express One Zone

考试里最常见的是：

S3 Standard：频繁访问、默认选择、高可用高性能。

S3 Intelligent-Tiering：访问模式不可预测时使用。它会根据访问频率自动调整层级，适合“不知道以后会不会访问”的对象。AWS 文档也说明，当不知道对象访问模式或访问模式会变化时，可以转到 S3 Intelligent-Tiering 以自动节省成本。 

S3 Standard-IA：不频繁访问，但需要跨 AZ 韧性。

S3 One Zone-IA：不频繁访问，只存一个 AZ，成本更低，但不适合要求 AZ 故障韧性的题。

S3 Glacier / Deep Archive：长期归档，取回不那么频繁。AWS 的 Glacier 页面说明，Glacier Flexible Retrieval 和 Glacier Deep Archive 面向低成本、安全可靠的归档场景。 

8. S3 Lifecycle 是什么？

S3 Lifecycle 是 S3 非常高频的考点。它的作用是：根据规则自动把对象移动到更便宜的存储层，或者在生命周期结束后删除对象。AWS 官方文档说明，可以用 S3 Lifecycle 规则将对象转换到更低成本的存储类别、归档或删除。 

例如：

第 0 天：上传日志到 S3 Standard
第 30 天：转到 S3 Standard-IA
第 90 天：转到 S3 Glacier
第 365 天：转到 S3 Glacier Deep Archive
第 2555 天：删除

考试判断：

访问模式不可预测 → S3 Intelligent-Tiering
明确 30 天后不访问 → S3 Lifecycle
长期保留最低成本 → Glacier Deep Archive
临时日志过期删除 → Lifecycle Expiration

你的文件中也明确写到：S3 Lifecycle -> Glacier Deep Archive/Glacier 适合“明确 1 个月后不访问且长期保留”或“14 天后归档分析数据”的场景。 

9. S3 Versioning 是什么？

S3 Versioning 是对象版本控制。开启后，同一个 key 被覆盖或删除时，S3 可以保留多个版本。AWS 官方文档说明，S3 Versioning 可以在一个 bucket 中保留对象的多个版本，从而恢复被意外删除或覆盖的对象。 

例如原来有：

report.pdf version 1

你上传新文件覆盖后：

report.pdf version 2

如果误删或误覆盖，可以恢复旧版本。它常和这些功能一起考：

防误删 → Versioning
防勒索/防篡改 → Versioning + Object Lock
跨区域容灾 → Versioning + Cross-Region Replication

10. S3 和静态网站

S3 可以托管静态网站，例如：

index.html
style.css
app.js
logo.png

如果只是 HTML/CSS/JavaScript/图片，不需要服务器动态渲染，那么 S3 static website hosting 是低成本方案。你的考点文件里也写到，纯 HTML/CSS/JS/图片站点最省钱方案是 S3 static website hosting。 

如果用户分布全球，通常加：

CloudFront + S3 Origin

这样用户访问时，会从边缘节点缓存内容，降低延迟。你的文件也把 CloudFront + S3 origin 总结为静态网站或静态资源全球低延迟访问方案。 

11. S3 和数据分析

S3 经常作为数据湖底座。日志、交易数据、点击流、CSV、JSON、Parquet 都可以放在 S3。然后用 Athena、Glue、EMR、Redshift Spectrum、QuickSight 等服务分析。

最典型考点：

日志已经在 S3
格式是 JSON / CSV / Parquet
临时 SQL 查询
最低运维

答案通常是：

Amazon Athena

你的文件中也总结：日志已经在 S3 且 JSON 格式，临时 SQL 查询选 Athena，避免搬到 Redshift/EMR。 

12. S3 和事件驱动

S3 可以在对象上传、删除等事件发生时触发后续处理。

常见架构：

User uploads image
        ↓
S3 Event Notification
        ↓
SQS
        ↓
Lambda
        ↓
Compress / Resize / Process

考试中如果看到：

图片上传后自动压缩
日志上传后自动处理
对象上传后触发无状态任务
需要持久缓冲

优先考虑：

S3 Event Notification → SQS → Lambda

你的文件也把这个链路列为上传对象后自动、持久、无状态处理的标准组合。 

13. S3 和迁移

如果本地有大量文件要迁移到 S3，方案取决于数据量和网络限制。

少量数据 → AWS CLI / SDK / DataSync
大量数据，网络足够 → AWS DataSync
几十 TB / 上百 TB，尽量少用网络 → Snowball Edge
本地 NFS/SMB 文件系统扩展到云 → S3 File Gateway
全球用户上传加速 → S3 Transfer Acceleration

你的文件中提到，几十 TB 级一次性迁移且尽量不用网络带宽，选 Snowball Edge → S3；全球站点高速上传到单一 S3 bucket，选 S3 Transfer Acceleration。 

14. S3 的加密

S3 支持服务端加密。AWS 文档说明，服务端加密是 S3 在 AWS 数据中心磁盘保存对象前加密对象，并在下载时解密。 

常见加密方式：

SSE-S3：S3 管理密钥
SSE-KMS：AWS KMS 管理密钥
SSE-C：客户自己提供密钥
Client-side encryption：客户端加密后再上传

考试里如果出现：

customer managed key
KMS key
key rotation
cross-Region same key

通常要看 SSE-KMS，如果强调跨 Region 还要看 multi-Region KMS key。你的文件也提醒，跨区域 S3 加密题要注意 “same KMS key” 和 “每个 Region 都存 key” 的 wording。 

15. S3 在 SAA-C03 中的核心判断

你可以这样记：

对象/图片/视频/日志/备份 → S3
静态网站 → S3 Static Website Hosting
全球静态内容低延迟 → CloudFront + S3
全球上传到单一 bucket → S3 Transfer Acceleration
私有子网访问 S3 → S3 Gateway VPC Endpoint
S3 上 JSON/CSV/Parquet 查询 → Athena
上传后自动处理 → S3 Event → SQS → Lambda
访问模式不可预测 → S3 Intelligent-Tiering
明确 N 天后归档 → S3 Lifecycle
长期最低成本归档 → Glacier Deep Archive
几十 TB 本地迁移且省带宽 → Snowball Edge
本地 SMB/NFS 扩展到云 → S3 File Gateway
组织内账号访问限制 → Bucket Policy + aws:PrincipalOrgID
EC2 访问 S3 → IAM Role，不要 Access Key

一句话总结：S3 是 AWS 的对象存储底座。它解决的是“海量、可靠、低成本地存放对象数据”的问题；围绕它再组合 CloudFront、Athena、Lifecycle、SQS、Lambda、KMS、VPC Endpoint、Snowball 等服务，形成静态网站、数据湖、归档、迁移、事件处理和安全访问架构。
