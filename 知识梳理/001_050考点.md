### 【考点 1】Amazon S3 存储策略与生命周期

**出现频率：15题（最高频）**  
**涉及题号：1, 6, 22, 23, 36, 38, 40, 44, 53, 63, 66, 72, 85, 88, 92**

#### 1.1 S3 数据传输加速

| 题号 | 场景 | 核心判断 | 正确答案 |
|------|------|----------|----------|
| Q1 | 多洲站点汇聚 500 GB/天到单一 S3 桶 | 高速网络+最小化操作复杂度 → **S3 Transfer Acceleration + 分段上传** | A |
| Q6 | 70 TB 本地 NFS 视频，最快迁移+最少网络带宽 | 大批量离线 → **Snowball Edge**（社区 86% 选 B） | B（社区）/ C（官方答案存争议） |
| Q88 | 3 TB 数据共享给欧洲合作伙伴，降低传输成本 | Requester Pays vs Cross-Region Replication | A（社区 44%）/ B（社区 46%）——**⚠️ 存在争议** |

**核心判断规则：**
- 网络良好 + 实时聚合 → S3 Transfer Acceleration
- 大批量 (≥几十 TB) + 最少带宽 → Snowball Edge
- 成本分摊给下载方 → Requester Pays

#### 1.2 S3 存储类选择

| 题号 | 场景 | 正确存储类 | 核心理由 |
|------|------|------------|----------|
| Q22 | 访问频率不可预测，需多 AZ 容灾 | **S3 Intelligent-Tiering** | 自动按访问频率分层，多 AZ |
| Q23 | 频繁访问 1 个月后极少访问，需永久保留 | **S3 Standard → Glacier Deep Archive（1个月后）** | 最低长期存储成本 |
| Q40 | 14 天热数据 + 归档 | **Kinesis Firehose → S3 + Lifecycle → Glacier** | 托管摄入 + 自动归档 |
| Q66 | 30 天频繁 + 4 年必须立即访问 | **S3 Standard → S3 Standard-IA（30天后）** | 需立即访问，不能用 Glacier |
| Q49 | 1 年内随机访问快速，超 1 年可延迟 | **S3 Standard → S3 Glacier Flexible Retrieval（1年后）** | 社区 69% 选 B |

**核心判断规则：**
- 访问模式不可预测 → Intelligent-Tiering
- 必须立即访问 → Standard / Standard-IA（绝不选 Glacier）
- 极少访问 + 允许延迟 + 最低成本 → Glacier Deep Archive
- 有明确冷热时间线 → Lifecycle Policy

#### 1.3 S3 数据保护

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q44 | 防止意外删除 | **Versioning + MFA Delete** | 双重保护；社区 98% AB |
| Q53 | 10 年不可删除（含 root 用户） | **Object Lock 合规模式（Compliance Mode）** | Compliance 模式连 root 也无法删除 |
| Q85 | 新文档上传后不可修改删除 | **S3 Versioning + S3 Object Lock** | 监管合规要求 |
| Q89 | 防止审计文档意外删除 | **Versioning + MFA Delete** | |

**核心判断规则：**
- Object Lock **Governance 模式**：有特定权限的用户可覆盖/删除
- Object Lock **Compliance 模式**：任何人（含 root）均无法删除，直到保留期结束
- MFA Delete 需额外身份验证才能删除版本

#### 1.4 S3 访问控制与 VPC 内部访问

| 题号 | 场景 | 正确方案 |
|------|------|----------|
| Q4 | EC2 访问 S3，不经公网 | **Gateway VPC Endpoint** |
| Q42 | EC2 通过 NAT 网关访问 S3，产生数据传输费 | **Gateway VPC Endpoint**（免区域内传输费） |
| Q72 | VPC 内 EC2 访问同 Region S3，降低费用 | **S3 VPC Gateway Endpoint** |
| Q91 | 不允许流量经互联网到 S3 | **S3 Gateway Endpoint** |
| Q92 | VPC 内 EC2 安全访问 S3 | **VPC Gateway Endpoint + Bucket Policy 限制 VPC** |
| Q3 | 限制 S3 仅 Organizations 内账户访问 | **aws:PrincipalOrgID 条件键** |

### 【考点 2】数据库高可用与读写分离

**出现频率：12题**  
**涉及题号：11, 13, 14, 30, 39, 61, 67, 69, 71, 86, 90, 93**

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q14 | 读多写少 + 自动扩展读容量 | **Aurora Multi-AZ + Aurora Auto Scaling + Aurora Replicas** | Aurora 读副本支持自动扩缩 |
| Q69 | EC2 Auto Scaling + 单 AZ Aurora，需高可用 | **Auto Scaling 多 AZ + Aurora Multi-AZ + RDS Proxy** | |
| Q90 | 脚本查询影响开发性能 | **创建 Read Replica，脚本只查只读副本** | 社区 95% 选 B |
| Q93 | 导出数据库到测试环境耗时长 | **Aurora 数据库克隆（Database Cloning）** | 近零拷贝，秒级完成 |
| Q30 | 月度测试用 48 小时，降成本 | **测试结束创建快照 → 终止实例 → 需要时恢复** | 停止实例有 7 天限制；RDS 不支持停止后立即重启节省成本 |
| Q71 | RPO 15 分钟，RTO 1 小时，DynamoDB | **DynamoDB 时间点恢复（PITR）** | PITR 提供连续备份 |
| Q67 | SQS 无重复消息但 RDS 出现重复记录 | **增加消息可见性超时（ChangeMessageVisibility）** | 防止消息被多次处理 |
| Q39 | RDS MySQL insert 慢，存储性能问题 | **改为 Provisioned IOPS SSD** | 社区 95% 选 A |
| Q13 | 多 Region RDS 凭证轮换 | **Secrets Manager 多 Region 副本 + 自动轮换** | |
| Q11 | EC2 连接 Aurora，消除本地凭证 | **AWS Secrets Manager + 自动轮换** | 社区 95% 选 A |
| Q61 | EC2 连接 RDS，不硬编码凭证 + 定期轮换 | **Secrets Manager + 自动轮换 + EC2 IAM 角色** | |
| Q86 | Web 服务器频繁访问 RDS MySQL，凭证频繁轮换 | **Secrets Manager + IAM 授权** | |

**核心判断规则：**
- 读多写少 → Read Replica（不是 Multi-AZ，Multi-AZ 用于故障转移）
- Multi-AZ → 高可用/故障转移（Standby 不接受读请求）
- 测试环境快速克隆数据库 → Aurora Database Cloning（非 mysqldump）
- 凭证管理 + 自动轮换 → Secrets Manager（Parameter Store 无原生自动轮换 RDS 凭证）

### 【考点 3】安全凭证与权限管理

**出现频率：10题**  
**涉及题号：3, 11, 13, 17, 26, 37, 61, 80, 86, 89**

| 题号 | 核心知识点 | 正确方案 |
|------|------------|----------|
| Q17 | EC2 访问 S3 的权限 | **IAM Role 附加到 EC2**（不是 IAM User/Group 直接附加） |
| Q37 | 远程管理 EC2，最小操作开销 | **IAM Role + Systems Manager Session Manager**（无需 SSH 密钥/堡垒机） |
| Q80 | 跨账户共享加密 AMI | **修改 launchPermission + 修改 KMS Key Policy 允许目标账户** |
| Q3 | 限制 S3 只允许 Org 内账户 | **aws:PrincipalOrgID 全局条件键** |
| Q26 | 监控 S3 配置变更 | **AWS Config 规则** |
| Q31 | 检查所有资源是否有标签 | **AWS Config 规则**（最低操作开销） |
| Q89 | 防止 S3 审计文档删除 | **Versioning + MFA Delete** |
| Q96 | IAM 策略效果分析 | 用户在 us-east-1，源 IP 10.100.100.254 时可以 terminate EC2 |

**核心判断规则：**
- EC2 访问 AWS 服务 → **IAM Role**（绝不是 IAM User 凭证存到实例）
- 检测资源配置变更/合规 → **AWS Config**（不是 CloudTrail，不是 Inspector）
- 记录 API 调用历史 → **AWS CloudTrail**
- 配置变更检测 vs API 审计：Config 跟踪配置状态，CloudTrail 记录 API 调用

### 【考点 4】消息队列与解耦架构

**出现频率：10题**  
**涉及题号：7, 8, 10, 18, 25, 33, 40, 45, 75, 87**

| 题号 | 场景 | 正确方案 | 核心理由 |
|------|------|----------|----------|
| Q7 | 突发 10 万/秒消息，多应用消费 | **SNS + SQS 扇出（Fan-out）** | SNS 推送，SQS 缓冲，解耦多消费者 |
| Q8 | 主协调服务器 + 计算节点，弹性扩展 | **SQS 队列 + Auto Scaling 基于队列深度扩展** | 社区 93% 选 B |
| Q10 | 电商订单必须按顺序处理 | **SQS FIFO 队列** | Standard 队列不保证顺序 |
| Q18 | S3 上传触发 Lambda 处理 | **S3 → SQS → Lambda**（持久化 + 无服务器） | |
| Q25 | Lambda 频繁写 Aurora 导致配额超限 | **两个 Lambda：接收 → SQS → 写库** | SQS 解耦，削峰填谷 |
| Q33 | 近实时共享交易数据 + 脱敏 + 存文档 DB | **Kinesis Data Streams → Lambda 脱敏 → DynamoDB** | |
| Q45 | SNS → Lambda 偶发网络故障丢数据 | **SNS → SQS（订阅）→ Lambda 读 SQS** | SQS 提供持久化重试 |
| Q75 | 多层 RESTful 微服务，事务丢失 | **API Gateway + Lambda + SQS 层间通信** | |
| Q87 | 数据库升级期间 Lambda 无法连接，数据丢失 | **SQS FIFO + 新 Lambda 轮询写库** | 社区 61% 选 D |
| Q98 | SQS 消息重复触发 Lambda → 重复邮件 | **增大可见性超时（Visibility Timeout）** | 社区 81% 选 C |

**核心判断规则：**
- 保证顺序 → SQS FIFO（Standard 队列不保证顺序且可能重复）
- 多消费者广播 → SNS（推送）；需要缓冲/重试 → SQS（拉取）
- 两者结合（Fan-out）→ SNS Topic + 多个 SQS 订阅
- 消息重复处理根因 → Lambda 超时 > 可见性超时 → 增大可见性超时
- 数据流实时处理大吞吐 → Kinesis Data Streams（非 SQS）

### 【考点 5】计算资源扩展与 Auto Scaling

**出现频率：9题**  
**涉及题号：8, 14, 20, 29, 52, 69, 70, 81, 84**

| 题号 | 核心知识点 | 正确方案 |
|------|------------|----------|
| Q8 | 基于队列深度扩展计算节点 | SQS + Auto Scaling（基于队列长度） |
| Q14 | 读副本自动扩展 | Aurora Auto Scaling + Aurora Replicas |
| Q20 | 快速克隆生产 EBS 到测试环境 | **EBS Fast Snapshot Restore**（启用后恢复性能无需预热） |
| Q29 | VoIP（UDP）多 Region 低延迟 + 自动故障转移 | **NLB + Global Accelerator**（支持 UDP；社区 78%） |
| Q52 | 标准文件系统结构 + 自动扩展 + 高可用 | **EC2 Auto Scaling + EFS**（EFS 自动扩展，跨 AZ） |
| Q69 | EC2 + Aurora 高可用，最小操作 | Auto Scaling 多 AZ + Aurora Multi-AZ + RDS Proxy |
| Q70 | NLB 无法检测 HTTP 错误 + 自动替换实例 | **替换为 ALB + HTTP 健康检查 + Auto Scaling 替换不健康实例** |
| Q81 | 无状态处理器并行扩展，作业持久化 | **SQS + Launch Template + Auto Scaling（基于队列深度）** |
| Q84 | 生产 24h 运行 vs 开发测试按需 | **生产用 Reserved，开发测试用 On-Demand** |

**核心判断规则：**
- NLB 支持 TCP/UDP（适合 VoIP）；ALB 支持 HTTP/HTTPS（支持 HTTP 健康检查）
- EBS 快照恢复默认需要预热（惰性加载）→ Fast Snapshot Restore 解决此问题
- EFS vs EBS：EFS 多 EC2 实例共享 + 自动扩展；EBS 绑定单实例
- 基于队列深度扩展是 Auto Scaling 与 SQS 结合的最佳实践

### 【考点 6】网络架构与 VPC 连接

**出现频率：9题**  
**涉及题号：4, 15, 19, 42, 43, 55, 68, 72, 73**

| 题号 | 场景 | 正确方案 |
|------|------|----------|
| Q4 | EC2 → S3，不经公网 | **Gateway VPC Endpoint** |
| Q15 | 流量检测与过滤（替换本地 Inspection Server） | **AWS Network Firewall** |
| Q19 | 第三方防火墙设备检测所有入站流量 | **Gateway Load Balancer（GWLB）**（社区 82%） |
| Q42 | EC2 经 NAT 访问 S3，避免传输费 | **S3 Gateway VPC Endpoint** |
| Q43 | 大量备份到 S3，减少公网带宽影响 | **AWS Direct Connect** |
| Q55 | 仅私有子网 EC2 可访问 RDS | **DB 安全组允许来自私有子网安全组的入站** |
| Q68 | 高可用低延迟 + 备用连接降成本 | **Direct Connect（主）+ VPN（备）** |
| Q72 | VPC 内访问 S3 降低传输费 | **S3 VPC Gateway Endpoint** |
| Q73 | 堡垒机 + 私有 EC2 访问控制 | **堡垒机 SG 允许公司外部 IP；应用实例 SG 允许堡垒机私有 IP** |

**核心判断规则：**
- S3/DynamoDB → Gateway Endpoint（免费，不经公网）
- 其他 AWS 服务 → Interface Endpoint（有费用）
- 第三方网络设备透明检测 → Gateway Load Balancer（GWLB）
- Direct Connect：专用线路，低延迟，稳定带宽
- VPN：加密隧道，成本低，但受公网影响

### 【考点 7】内容分发与全球加速

**出现频率：7题**  
**涉及题号：12, 21, 29, 38, 59, 83, 84**

| 题号 | 场景 | 正确方案 | 要点 |
|------|------|----------|------|
| Q12 | 静态 (S3) + 动态 (ALB) 统一加速 | **CloudFront（双 Origin：S3 + ALB）** | 一个 Distribution 两个 Origin |
| Q21 | 单品秒杀，百万请求/小时，毫秒延迟 | **S3 静态 + CloudFront + API Gateway + Lambda + DynamoDB** | 全无服务器架构 |
| Q38 | 全球访问静态 S3 网站，降延迟 | **CloudFront + Route 53** | 最低成本；Transfer Acceleration 面向上传非下载 |
| Q29 | VoIP 多 Region 低延迟 + UDP | **NLB + AWS Global Accelerator** | GA 支持 TCP/UDP；CloudFront 不适合 UDP |
| Q59 | 30 TB/天点击流数据处理分析 | **Kinesis Data Streams → Firehose → S3 → Redshift** | |
| Q83 | 欧洲新用户，后端留在美国 | **CloudFront 自定义源指向本地服务器** | 最快方案，无需迁移后端 |

**核心判断规则：**
- CloudFront：HTTP/HTTPS 缓存加速，适合静态+动态内容
- Global Accelerator：TCP/UDP 全协议加速，适合非 HTTP 场景（VoIP、游戏）
- CloudFront 不支持 UDP → VoIP 必须用 Global Accelerator

### 3.1 高频组合模式

```
模式一：无服务器事件驱动
S3 上传事件 → SQS → Lambda → DynamoDB/S3
涉及题号：Q18, Q63, Q94

涉及题号：Q7, Q45

模式三：全球静态网站加速
S3 (静态资源) + CloudFront + Route 53 + API Gateway + Lambda (API)
涉及题号：Q21, Q38, Q83

模式四：混合云文件访问
本地文件服务器 ← File Gateway → S3/FSx (云端)
涉及题号：Q9, Q54, Q64

模式五：数据流实时处理管道
数据源 → Kinesis Data Streams → Lambda/Firehose → S3 → Redshift/Athena
涉及题号：Q33, Q40, Q59, Q77
```

### 3.2 容易混淆的服务对比

| 维度 | 选项 A | 选项 B | 判断依据 |
|------|--------|--------|----------|
| 顺序保证 | SQS FIFO | SQS Standard | FIFO 保证顺序，Standard 不保证 |
| 流量检测 | Network Firewall | GuardDuty | Firewall 过滤流量；GuardDuty 检测威胁情报 |
| 配置合规 | AWS Config | CloudTrail | Config 检测"配置状态"；CloudTrail 记录"API操作" |
| 文件共享 | EFS (Linux/NFS) | FSx for Windows (SMB/AD) | 按操作系统和协议选择 |
| 大数据 | Kinesis (实时流) | EMR (批处理) | 按实时 vs 批次选择 |
| 数据可视化 | QuickSight | Athena | QuickSight 是 BI 可视化；Athena 是 SQL 查询 |
| 访问日志分析 | Athena (S3 + SQL) | CloudWatch Logs | 按存储位置选择；Athena 适合大量 S3 日志 |
| 全球加速 | CloudFront (HTTP缓存) | Global Accelerator (TCP/UDP) | HTTP 内容用 CF；非 HTTP 用 GA |
| 凭证管理 | Secrets Manager (自动轮换) | Parameter Store (手动) | 需要自动轮换数据库凭证 → Secrets Manager |
| Object Lock | Compliance（连root不能删） | Governance（有权限可覆盖） | 监管合规要求 → Compliance 模式 |

### 3.3 "最少操作开销"判断框架

当题目要求"LEAST operational overhead"时，优先级：

1. **全托管无服务器**（Lambda, Fargate, DynamoDB, Aurora Serverless）
2. **托管 PaaS**（RDS, ElastiCache, OpenSearch）
3. **自动化策略**（Lifecycle Policy, Auto Scaling, Config Rules）
4. **需自管基础设施**（EC2, 自部署软件）→ 通常是错误选项
