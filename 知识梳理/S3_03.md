# Amazon S3 详细说明

---

## 一、存储层级（Storage Classes）

S3 提供多种存储类型，核心逻辑是：**访问频率越低 → 存储费用越低 → 检索费用越高、延迟越长**。

### 1.1 完整层级体系

```
访问频率：高 ────────────────────────────────────────── 低
存储成本：贵 ────────────────────────────────────────── 便宜

Standard → Intelligent-Tiering → Standard-IA → One Zone-IA 
→ Glacier Instant → Glacier Flexible → Glacier Deep Archive
```

### 1.2 各层详细对比

| 存储类型 | 跨AZ冗余 | 检索延迟 | 最低存储时长 | 最低对象大小计费 | 核心适用场景 |
|---|---|---|---|---|---|
| **Standard** | ✅ 3+ AZ | 毫秒 | 无 | 无 | 热数据、频繁访问 |
| **Intelligent-Tiering** | ✅ 3+ AZ | 毫秒 | 无 | 128KB | 访问模式不可预测 |
| **Standard-IA** | ✅ 3+ AZ | 毫秒 | **30天** | **128KB** | 不常访问但需快速检索 |
| **One Zone-IA** | ❌ 单AZ | 毫秒 | 30天 | 128KB | 可重新生成的非关键数据 |
| **Glacier Instant Retrieval** | ✅ 3+ AZ | 毫秒 | **90天** | 128KB | 归档但偶尔需立即访问 |
| **Glacier Flexible Retrieval** | ✅ 3+ AZ | 分钟～12小时 | 90天 | 40KB | 传统归档，不急用 |
| **Glacier Deep Archive** | ✅ 3+ AZ | **12小时** | **180天** | 40KB | 超长期合规存档，最便宜 |

### 1.3 Intelligent-Tiering 内部机制

Intelligent-Tiering 不是单一层，它内部自动在多个访问层之间切换：

```
连续30天未访问 → 从 Frequent Access 移到 Infrequent Access
连续90天未访问 → 移到 Archive Instant Access（可选开启）
连续180天未访问 → 移到 Deep Archive Access（可选开启）
一旦被访问 → 自动回到 Frequent Access
```

**费用结构**：每个对象额外收取监控费（约 $0.0025/1000对象/月），小对象（< 128KB）不会被移到低频层，但仍收监控费——因此小对象多的场景用 Intelligent-Tiering 反而不划算。

### 1.4 隐形成本陷阱（考试高频）

**最低存储时长**的含义：若数据存入后提前删除或转移，仍按最低时长收费。

实例说明：
- 数据存入 Standard-IA，第 10 天删除 → **仍按 30 天计费**
- 数据存入 Glacier Deep Archive，第 60 天删除 → **仍按 180 天计费**

**最低对象大小计费**：Standard-IA 对小于 128KB 的对象按 128KB 计费。大量小文件转 IA 层实际更贵，正确做法是先合并（S3 Batch Operations）再转层。

## 二、生命周期策略（Lifecycle Policy）

生命周期策略是 S3 自动化成本管理的核心工具，本质是定义「数据随时间流逝的归宿」。

### 2.1 两类操作

**Transition Actions（转层）**：将对象从一个存储类型移动到另一个。

**Expiration Actions（过期删除）**：在指定时间后自动删除对象。

### 2.2 转层方向限制（单向，不可逆）

```
Standard
   ↓（可以）
Intelligent-Tiering / Standard-IA / One Zone-IA
   ↓（可以）
Glacier Instant Retrieval
   ↓（可以）
Glacier Flexible Retrieval
   ↓（可以）
Glacier Deep Archive
```

**不能反向转层**（如从 Glacier 转回 Standard），恢复数据需要先 Restore（临时解冻），然后手动复制到 Standard bucket。

### 2.3 典型生命周期配置示例

**场景 A：日志归档（题 #23 对应）**
```
Day 0      → 上传到 Standard（热数据，频繁查询）
Day 30     → Transition 到 Standard-IA（查询减少）
Day 90     → Transition 到 Glacier Flexible Retrieval
Day 365    → Transition 到 Glacier Deep Archive（合规留存）
Day 2555   → Expiration（7年后删除）
```

**场景 B：短期热数据直接归档（题 #40 对应）**
```
Day 0  → Kinesis Firehose 写入 Standard
Day 14 → Transition 到 Glacier（不再需要快速访问）
```

**场景 C：测试环境临时文件**
```
Day 0  → 上传测试数据
Day 7  → Expiration（自动删除，不需要手动清理）
```

### 2.4 Multipart Upload 残留清理

上传大文件时分片未完成会产生「未完成的 Multipart Upload」残留，持续计费但不可用。

生命周期策略可配置：`AbortIncompleteMultipartUpload after N days`，自动清理残留分片，这是一个常被忽视的成本优化点。

### 2.5 版本控制（Versioning）与生命周期联动

开启 Versioning 后，每次覆盖上传会生成新版本，旧版本保留。生命周期可以针对「非当前版本（Noncurrent versions）」单独设置：

```
当前版本    → Standard（保持热存储）
非当前版本  → 30天后转 Glacier Deep Archive
删除标记    → 90天后 Expiration（清理 delete markers）
```

## 三、访问控制（Access Control）

S3 的访问控制是一个多层叠加体系，理解层次关系是关键。

### 3.1 控制层次总览

```
层级1：Block Public Access（账号/Bucket 级别的总开关）
   ↓
层级2：Bucket Policy（Resource-based，JSON格式，细粒度控制）
   ↓
层级3：IAM Policy（Identity-based，控制哪个主体能做什么）
   ↓
层级4：ACL（Access Control List，对象级，已基本废弃）
```

**最终权限 = 所有层级的交集**，任何层级的显式 Deny 优先于所有 Allow。

### 3.2 Block Public Access（最重要的安全开关）

这是 2018 年 AWS 针对 S3 数据泄露事故引入的防护机制，分四个独立开关：

| 设置项 | 作用 |
|---|---|
| BlockPublicAcls | 拒绝设置公开 ACL |
| IgnorePublicAcls | 忽略已有的公开 ACL |
| BlockPublicPolicy | 拒绝设置公开 Bucket Policy |
| RestrictPublicBuckets | 限制公开 Bucket Policy 的效果 |

**最佳实践**：在账号级别全部开启，对需要公开的 Bucket 单独例外放开。

### 3.3 Bucket Policy 详解

Bucket Policy 是 JSON 格式的 Resource-based Policy，挂在 Bucket 上。

**基础结构**：

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",          // Allow 或 Deny
      "Principal": "*",           // 谁（账号/服务/所有人）
      "Action": "s3:GetObject",   // 做什么操作
      "Resource": "arn:aws:s3:::my-bucket/*",  // 哪个资源
      "Condition": {}             // 附加条件（可选）
    }
  ]
}
```

**关键条件键（Condition Keys）**：

| 条件键 | 用途 | 典型场景 |
|---|---|---|
| `aws:PrincipalOrgID` | 限定 AWS Organization 内账号 | 组织内跨账号共享（题#3） |
| `aws:SourceIp` | 限定来源 IP 或 CIDR | 只允许公司 IP 访问 |
| `aws:SecureTransport` | 强制 HTTPS | 拒绝 HTTP 明文访问 |
| `s3:prefix` | 限定对象前缀 | 部门级数据隔离 |
| `aws:PrincipalAccount` | 限定特定账号 | 跨账号精细控制 |

**强制 HTTPS 的标准写法**（合规场景必备）：

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"],
  "Condition": {
    "Bool": {
      "aws:SecureTransport": "false"
    }
  }
}
```

### 3.4 IAM Policy 与 Bucket Policy 的协作

**同账号内访问 S3**：IAM Policy 或 Bucket Policy 任一 Allow 即可（两者取并集）。

**跨账号访问 S3**：必须两边同时 Allow——被访问账号的 Bucket Policy 要允许，访问方账号的 IAM Policy 也要允许（两者取交集）。

**EC2 访问 S3 的正确姿势**（题 #17）：
```
EC2 实例 → 挂载 IAM Role（Instance Profile）→ Role 有 S3 权限
```
不应使用 IAM User 的长期 Access Key 写在代码或环境变量中，这是安全反模式。

### 3.5 CloudFront + S3 的访问控制（OAC/OAI）

当 S3 作为 CloudFront Origin 时，不应将 Bucket 设为公开，而是用 CloudFront 专属身份控制访问：

**Origin Access Control（OAC，新版推荐）**：
```
用户 → CloudFront → S3（Bucket Policy 只允许 CloudFront Service Principal）
```

Bucket Policy 示例：
```json
{
  "Principal": {
    "Service": "cloudfront.amazonaws.com"
  },
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudfront::account-id:distribution/distribution-id"
    }
  }
}
```

**OAI（旧版）vs OAC（新版）区别**：OAC 支持 SSE-KMS 加密的 S3 Bucket，OAI 不支持——这是考题中区分两者的关键。

### 3.6 VPC 内访问控制：Gateway VPC Endpoint

私有子网中的 EC2 通过 Gateway VPC Endpoint 访问 S3：

```
私有子网 EC2 → 路由表（目标 S3 prefix list）→ Gateway Endpoint → S3
```

Bucket Policy 可进一步限制只允许来自特定 VPC Endpoint 的访问：

```json
{
  "Condition": {
    "StringEquals": {
      "aws:SourceVpce": "vpce-xxxxxxxxx"
    }
  }
}
```

这样即使 IAM 权限存在，来自 Internet 的请求也会被拒绝，形成双重防护。

## 四、与其他服务的集成组合

### 4.1 S3 + CloudFront（内容分发）

**适用场景**：静态资源全球低延迟访问、静态网站托管。

```
用户（全球）→ CloudFront Edge Location（缓存命中直接返回）
                    ↓（缓存未命中）
             S3 Bucket（Origin，私有）
```

**关键配置点**：
- Cache-Control 头控制缓存时长
- CloudFront Invalidation 主动刷新缓存
- 自定义域名 + ACM 证书实现 HTTPS
- 多 Origin 配置：`/static/*` 路由到 S3，`/api/*` 路由到 ALB

### 4.2 S3 + Athena（数据分析）

**适用场景**：对 S3 上的结构化/半结构化数据进行 ad-hoc SQL 查询。

```
数据源 → S3（JSON/CSV/Parquet/ORC）→ Athena（Serverless SQL）→ 结果输出到 S3
                                          ↓
                                     QuickSight（可视化）
```

**性能与成本优化**：

| 优化手段 | 效果 |
|---|---|
| 数据转 Parquet/ORC 列式格式 | 减少扫描量 60-90% |
| 按时间/地区 Partition | 只扫描相关分区 |
| 压缩（Snappy/GZIP） | 减少存储和扫描成本 |
| 使用 Glue Data Catalog | 统一元数据管理，Athena 直接引用 |

### 4.3 S3 + Lambda（事件驱动处理）

**模式 A：直接触发（低并发场景）**
```
S3 上传对象 → S3 Event Notification → Lambda 直接触发
```

**模式 B：SQS 缓冲（高并发、不丢失场景，考试优选）**
```
S3 上传对象 → S3 Event → SQS 队列 → Lambda（批量消费）
```

**模式 C：EventBridge（复杂路由场景）**
```
S3 上传对象 → EventBridge → 多个目标（Lambda/StepFunctions/SNS）
```

**选择逻辑**：
- 简单低并发 → 直接触发 Lambda
- 需要持久缓冲、重试、不丢失 → SQS → Lambda
- 需要基于内容路由或触发多个下游 → EventBridge

### 4.4 S3 + Kinesis Firehose（流式数据摄取）

**适用场景**：持续大量数据写入 S3，无需管理基础设施。

```
数据源（IoT/日志/告警）→ Kinesis Data Firehose 
  → 可选：Lambda 转换/过滤
  → S3（自动按时间前缀组织）
  → Lifecycle → Glacier（归档）
```

Firehose 自动处理：分片聚合、格式转换（JSON → Parquet）、压缩、错误重试，是「数据落 S3」最低运维的方案。

### 4.5 S3 + Glacier（生命周期归档）

**这是 S3 与自身存储层级的集成**，通过 Lifecycle Policy 自动实现：

```
热数据（Standard）→ [30天] → 温数据（Standard-IA）
                 → [90天] → 冷数据（Glacier Flexible）
                 → [365天] → 冷归档（Glacier Deep Archive）
                 → [7年] → Expiration（删除）
```

### 4.6 S3 + Snowball Edge（大规模迁移）

**适用场景**：网络带宽受限时的大批量数据上云。

```
本地数据中心 → 写入 Snowball Edge 设备（可达 80TB/台）
             → 物理快递给 AWS
             → AWS 将数据导入指定 S3 Bucket
```

迁移完成后可配合 S3 Lifecycle 立即开始归档冷数据。

### 4.7 S3 File Gateway（混合云存储）

**适用场景**：本地应用无法改造、需要继续使用 NFS/SMB 协议，同时将数据实际存储在云端。

```
本地应用（SMB/NFS）→ S3 File Gateway（本地部署，缓存热数据）
                         ↓（异步同步）
                    S3 Bucket → Lifecycle → Glacier
```

**与 Storage Gateway 其他类型对比**：

| 类型 | 协议 | 后端存储 | 适用场景 |
|---|---|---|---|
| S3 File Gateway | NFS/SMB | S3 | 文件共享扩展到云 |
| FSx File Gateway | SMB | Amazon FSx | Windows 文件服务器工作负载 |
| Volume Gateway | iSCSI | EBS Snapshot | 块存储备份 |
| Tape Gateway | VTL | Glacier | 替代物理磁带备份 |

### 4.8 S3 + QuickSight（BI 可视化）

```
S3（CSV/Parquet 数据）→ Athena（SQL 转换）→ QuickSight（Dashboard）
                                                    ↓
                                           QuickSight Users/Groups（访问控制）
```

QuickSight 可以直接连接 S3（通过 Manifest 文件），也可以通过 Athena 作为中间层，后者更灵活且支持复杂查询。

### 4.9 S3 + Replication（跨区域/跨账号复制）

**两种复制模式**：

| 模式 | 全称 | 适用场景 |
|---|---|---|
| CRR | Cross-Region Replication | 灾备、降低延迟、合规数据主权 |
| SRR | Same-Region Replication | 日志聚合、测试环境复制生产数据 |

**前提条件**：必须开启 Versioning（两端 Bucket 都需要）。

**与加密的联动（题 #36）**：跨 Region 复制 SSE-KMS 加密对象时，若要求目标 Region 使用「同一个 KMS 密钥」，需使用 **Multi-Region KMS Key**。

## 五、综合架构速查

| 业务场景 | 推荐架构组合 | 关键判断词 |
|---|---|---|
| 全球静态网站 | S3 + CloudFront + Route 53 | `static`, `global`, `HTTPS` |
| Serverless Web 应用 | S3 + CloudFront + API GW + Lambda + DynamoDB | `millions requests`, `no server` |
| 数据湖查询 | S3 + Glue + Athena + QuickSight | `ad-hoc SQL`, `visualization` |
| 流式数据归档 | Kinesis Firehose + S3 + Lifecycle | `ingest`, `no infrastructure`, `archive` |
| 图片上传处理 | S3 Event + SQS + Lambda | `upload`, `process`, `durable` |
| 大量数据上云 | Snowball Edge → S3 | `TB级`, `limited bandwidth` |
| 混合云文件存储 | S3 File Gateway + S3 Lifecycle | `on-premises NFS/SMB` |
| 合规长期归档 | S3 + Lifecycle → Glacier Deep Archive | `compliance`, `long-term retention` |
| 私网访问 S3 | Gateway VPC Endpoint | `private subnet`, `no internet`, `free` |
| 跨区复制+加密 | CRR + Multi-Region KMS Key | `cross-region`, `same KMS key` |

## 六、思维扩展

**1. 对象存储的「不可变性」是架构设计的关键约束**
S3 不支持随机写、不支持文件锁、不支持 append（只能整体覆盖），这决定了它天然适合「写一次、读多次」的场景（日志、备份、媒体），而不适合数据库数据文件、实时读写的应用数据。理解这一限制，就能快速排除「将数据库直接放 S3」类的错误选项。

**2. 生命周期策略的本质是「时间换空间换金钱」**
AWS 定价模型下，数据越冷越便宜，但检索越贵、延迟越长。生命周期策略的本质是在「访问速度需求」和「存储成本」之间做时间维度的动态权衡。考题中的最优解永远是：根据业务对数据检索时间的容忍度，匹配合适的存储层。

**3. S3 安全的「纵深防御」思想**
S3 的访问控制不是单点防线，而是 Block Public Access → Bucket Policy → IAM Policy → VPC Endpoint Policy → KMS 加密的多层防御。每一层独立失效都不会导致整体安全崩溃。考试中如果题目强调「多重保护」或「合规」，答案往往需要组合多个控制层，而不是单一措施。

**4. 「最低运维」倾向的背后是 AWS 的商业逻辑**
SAA-C03 大量题目的正确答案都指向 Serverless 和托管服务（Athena、Firehose、Lambda），这背后是 AWS 的产品战略——推动客户使用更高附加值的托管服务，减少自管基础设施。理解这一逻辑，遇到模糊题目时优先选「运维最少」的方案，胜率显著提升。
